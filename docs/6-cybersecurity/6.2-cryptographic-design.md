## 1. Purpose

This document specifies the cryptographic primitives selected for ParkSense, the rationale for each selection, the forbidden algorithm list, and the complete key management lifecycle. It is the implementation-level companion to the high-level controls described in [[4.6-security-architecture]].

---

## 2. Cryptographic Hardware Resources

The STM32U585AII6Q provides hardware accelerators for all ParkSense cryptographic operations. Software implementations are not used.

| Accelerator | STM32U5 Peripheral | Operations | Hardware Resistance |
|-------------|-------------------|-----------|---------------------|
| AES-256/128 | `AES` (HW) | AES-128-CCM encryption/decryption | Side-channel: HW masking, auto-masking option |
| SHA-256 | `HASH` (HW) | Image digest, key derivation | DMA-fed, no byte copy to/from CPU |
| ECDSA P-256 | `PKA` (Public Key Accelerator) | Signature generation/verification | HW arithmetic; no software big-integer library |
| True RNG | `RNG` (HW) | Key generation, nonces | FIPS 140-2 compliant; conditioning function included |
| On-the-fly decryption | `OTFDEC` | Decrypt flash at read time (future use) | Transparent to CPU |

**All cryptographic operations are performed in the Secure World.** The NS application invokes crypto via secure gateway functions that never expose key material to the NS address space.

## 3. Algorithm Selection

### 3.1 Symmetric Encryption — AES-128-CCM

| Parameter | Value | Rationale |
|-----------|-------|-----------|
| Algorithm | AES-128 | 128-bit keys are sufficient for the threat model (NIST approved through 2030+); matches Zigbee 3.0 mandatory algorithm |
| Mode | CCM (Counter with CBC-MAC) | Zigbee 3.0 mandates CCM; provides both confidentiality and authentication in a single pass |
| Key size | 128 bits | Mandatory for Zigbee 3.0 compliance; 128-bit provides 2^128 brute-force resistance (computationally infeasible) |
| Nonce size | 13 bytes | IEEE 802.15.4 CCM* nonce format: 8-byte source address + 5-byte frame counter |
| MIC size | 4 bytes | Zigbee NWK layer default; provides 2^32 ≈ 1-in-4-billion forgery probability per frame |
| Hardware | STM32U585 AES peripheral | Eliminates timing side-channels present in software AES implementations |

**Why CCM and not GCM?**
- Zigbee 3.0 mandates CCM* — no protocol choice here
- CCM is suitable for short messages (12-byte CPM payload) and resource-constrained MCUs
- GCM is preferred for high-throughput TLS but has higher implementation complexity on embedded

### 3.2 Asymmetric Signing — ECDSA P-256

| Parameter | Value | Rationale |
|-----------|-------|-----------|
| Algorithm | ECDSA (Elliptic Curve Digital Signature Algorithm) | Smaller key + signature than RSA at equivalent security; hardware-accelerated on STM32U585 |
| Curve | NIST P-256 (secp256r1) | NIST approved, FIPS 186-4 compliant; 128-bit equivalent security; widely supported |
| Hash | SHA-256 | Matched to curve security level; hardware-accelerated |
| Signature size | 64 bytes (r || s) | Fits in 256-byte image header (64 bytes allocated) |
| Key size | 256-bit private, 512-bit public (uncompressed) | P-256 standard |

**Why ECDSA P-256 and not Ed25519?**
- STM32U585 PKA explicitly supports NIST P-256; Ed25519 (Edwards curve) requires different arithmetic not accelerated by the PKA peripheral
- Both provide ~128-bit security; P-256 is the pragmatic choice on this hardware
- Future hardware (STM32H5, STM32U5 with SAES) may support Ed25519 in hardware

**Why not RSA-2048?**
- RSA-2048 signature = 256 bytes — exactly the entire image header budget
- RSA private key operations are 10–100× slower on constrained MCUs
- ECDSA P-256 signature verification completes in ~10 ms on STM32U585 at 160 MHz vs. ~300 ms for RSA-2048

### 3.3 Hash — SHA-256

| Parameter | Value | Rationale |
|-----------|-------|-----------|
| Algorithm | SHA-256 | NIST approved; 256-bit digest; 128-bit collision resistance; hardware-accelerated |
| Usage | Image integrity (digest in image header), ECDSA pre-hash | Standard pairing with ECDSA P-256 |

**Why not SHA-384 or SHA-512?**
- SHA-256 provides 128-bit second pre-image resistance — matched to the 128-bit security level of P-256 and AES-128
- Using a stronger hash with weaker primitives provides no overall benefit
- SHA-256 is faster (fewer rounds) and the STM32U585 HASH peripheral is optimized for it

### 3.4 TLS (Gateway → Server)

| Parameter | Value | Rationale |
|-----------|-------|-----------|
| Protocol | TLS 1.2 minimum | TLS 1.3 is preferred but mbedTLS version on EMW3080B may not support it; TLS 1.2 with strong cipher suites is acceptable |
| Cipher suite | `TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256` (preferred) | Forward secrecy via ECDHE; AES-128-GCM for transport; widely supported |
| Certificate validation | Server X.509 certificate validated against embedded CA certificate | Prevents MITM; CA cert embedded in gateway firmware at build time |
| Client authentication | Not required (server recognizes gateway by pre-shared session token, future) | Phase 1 simplification; Phase 2 adds mutual TLS |

### 3.5 True Random Number Generator (TRNG)

| Parameter | Value |
|-----------|-------|
| Source | STM32U585 `RNG` peripheral — hardware noise source (ring oscillator jitter) |
| Conditioning | FIPS SP 800-90B compliant noise source conditioning |
| Output | 32-bit words; used for: nonce generation, session key randomness, CRC seed |
| Health tests | Startup test (FIPS) + continuous online health test |

## 4. Forbidden Algorithm List

The following algorithms are prohibited in ParkSense firmware. Any use triggers a MISRA static analysis violation or code review rejection.

| Algorithm | Category | Reason Forbidden |
|-----------|----------|-----------------|
| MD5 | Hash | Collision found (1996); NIST deprecated |
| SHA-1 | Hash | Practical collision attack (2017); NIST deprecated for digital signatures |
| DES, 3DES | Symmetric | Effective key size ≤ 112 bits; NIST deprecated as of 2023 |
| AES-ECB | Cipher mode | Deterministic — identical plaintext → identical ciphertext; reveals patterns |
| RC4 | Stream cipher | Biases in keystream; prohibited by RFC 7465 |
| RSA < 2048-bit | Asymmetric | Factoring attacks feasible below 2048 bits |
| Custom/homebrew crypto | Any | Avoid "roll your own" cryptography — NIST vetted algorithms only |
| `rand()` from C stdlib | PRNG | Not cryptographically secure; use STM32U585 `RNG` hardware |

## 5. Node Commissioning and Key Enrollment

### 5.1 Install Code (Pre-Shared Key Derivation)

Each IoT node is provisioned with a unique **Install Code** at manufacturing time. The Install Code is a 16-byte random value generated by the STM32U585 TRNG and stored in the CONFIG_FLASH sector.

```
Install Code (16 B, random, unique per device)
  │
  ▼
AES-MMO hash (Matyas-Meyer-Oseas, Zigbee standard)
  │
  ▼
Join Link Key (16 B, unique per device)
  │
  ▼
Used to encrypt NWK Key delivery from Trust Center (gateway) to joining node
```

**Why Install Code and not a default join key?**
- A default join key (e.g., `Trust Center Link Key = "ZigBeeAlliance09"`) allows any device to join any Zigbee 3.0 network
- Install Code ensures that only a node whose Install Code has been registered with the Trust Center can join
- Satisfies SYS-S-013 (secure commissioning) and ETSI EN 303 645 §5.1 (no universal defaults)

### 5.2 Commissioning Workflow

```
Factory Provisioning:
  1. Generate Install Code via STM32U585 TRNG
  2. Store Install Code in CONFIG_FLASH
  3. Print Install Code + EUI-64 on device label (QR code)

Field Deployment:
  4. Operator scans node QR code with provisioning tool
  5. Provisioning tool registers (EUI-64, Install Code) with gateway Trust Center
  6. Node powers on → scans for PAN → sends Association Request
  7. Trust Center looks up Install Code by EUI-64 → derives Join Link Key
  8. Trust Center encrypts NWK Key with Join Link Key → sends to node
  9. Node decrypts → installs NWK Key → joins PAN
```

### 5.3 Network Key (NWK Key)

| Parameter | Value |
|-----------|-------|
| Type | Zigbee Network Key |
| Size | 128 bits (AES-128) |
| Generation | Generated by gateway Trust Center using STM32U585 TRNG |
| Distribution | Encrypted with each node's Join Link Key |
| Storage (node) | SRAM3_S (Secure World) at runtime; not persisted to flash |
| Storage (gateway) | SRAM3_S; also registered with STM32WB Zigbee stack |
| Rotation | Manual rotation via provisioning tool on key compromise; scheduled rotation in Phase 2 |

## 6. Key Management Lifecycle

### 6.1 Key Inventory

| Key | Algorithm | Bit Length | Storage | Generated By | Rotation |
|-----|-----------|-----------|---------|-------------|---------|
| ECDSA Private Signing Key | ECDSA P-256 | 256 | GitHub Actions Secret (never on device) | HSM or `openssl` on air-gapped machine | On compromise; or yearly |
| ECDSA Public Key | ECDSA P-256 | 512 (uncompressed) | Bootloader flash (Secure sector) | Paired with private key | When private key rotates |
| Zigbee NWK Key | AES-128 | 128 | SRAM3_S (runtime) + Trust Center DB | STM32U585 TRNG | On compromise |
| Per-device Install Code | Random | 128 | CONFIG_FLASH + provisioning DB | STM32U585 TRNG at provisioning | On node replacement |
| Per-device Join Link Key | AES-128 (derived) | 128 | Not stored — derived on demand from Install Code | Derived by Trust Center (AES-MMO) | Derived fresh each join |
| WiFi PSK | WPA2 passphrase | Variable | SRAM3_S (runtime); not in firmware binary | Operator-configured at gateway deployment | On AP change |
| TLS CA Certificate | X.509 (RSA or ECDSA) | Variable | Gateway firmware flash (NS region) | Server certificate authority | On expiry or compromise |

### 6.2 Key Generation Rules

1. **All symmetric keys** are generated using the STM32U585 hardware TRNG with full entropy conditioning.
2. **ECDSA private key** is generated offline, outside the firmware build environment. Never in a VM with network access.
3. **Private key must never touch** a networked machine, source repository, build artifact, or device flash.
4. **Key generation ceremony** (for production) requires two-person authorization, air-gapped machine, and HSM storage.

### 6.3 Key Storage Rules

| Key | Allowed Storage Locations | Forbidden Locations |
|-----|--------------------------|---------------------|
| ECDSA private key | HSM, GitHub Actions Secret, encrypted key file (offline) | Source code, binary, flash, SRAM (NS), email |
| ECDSA public key | Bootloader flash (Secure sector), source code (public) | — (public key is not secret) |
| NWK Key | SRAM3_S (runtime only) | NS SRAM, flash (unencrypted), config file |
| Install Code | CONFIG_FLASH (only acceptable flash location) | NS SRAM at runtime, source code |
| WiFi PSK | SRAM3_S (runtime only) | Source code, NS flash, binary |

### 6.4 Key Rotation and Revocation

| Key | Rotation Trigger | Revocation Procedure |
|-----|-----------------|---------------------|
| ECDSA signing key | Compromise suspected; annual review | Generate new keypair → build new bootloader with new public key → OTA all devices → revoke old key |
| Zigbee NWK Key | Compromise suspected; node removed from service | Trust Center broadcasts new NWK key to all active nodes (Zigbee key update command) |
| Install Code | Node physical replacement | Deregister old (EUI-64, Install Code) from Trust Center DB; provision new node |
| WiFi PSK | AP password change | Gateway re-provisioned (physical access or secure admin channel) |

### 6.5 Anti-Rollback (Signing Key Rotation Sequence)

When the ECDSA signing key is rotated, existing devices in the field must be migrated:

```
1. Generate new ECDSA keypair (P-256)
2. Build new bootloader embedding new public key
3. Sign new bootloader with OLD private key → verify using old public key on current devices ✓
4. OTA-distribute new bootloader
5. New bootloader installs → contains new public key
6. All subsequent OTA images are signed with NEW private key
7. OLD private key is revoked / destroyed
8. MIN_VERSION is incremented in new bootloader to prevent rollback to old-bootloader images
```
