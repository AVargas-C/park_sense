## 1. Purpose

This document provides the full threat model for ParkSense — beyond the 10-row summary in 4.6. It defines adversary profiles, enumerates the complete attack surface, builds attack trees for the highest-risk paths, and scores each threat using a qualitative risk matrix. The output drives control selection in [[6.2-cryptographic-design]] and test case design in [[6.4-penetration-test-plan]].

## 2. Adversary Profiles

| Profile | Motivation | Capability | Physical Access | Likely Attacks |
|---------|-----------|-----------|----------------|---------------|
| **Opportunistic vandal** | Disruption, curiosity | Low — commodity tools only | Direct (parking lot) | Physical damage, sensor relocation, RF jamming with off-the-shelf devices |
| **Competitor / business intelligence** | Occupancy data for competitive analysis | Medium — software-defined radio, scripting | Proximity (within RF range) | RF eavesdropping, passive traffic analysis |
| **Disgruntled employee / insider** | Sabotage, data theft | Medium-High — knows system details | Direct + network | Firmware modification via JTAG, rogue node injection, WiFi credential theft |
| **Automated scanner / botnet** | Botnet recruitment, lateral movement | Low-Medium — automated tools | Network only | Port scanning, credential stuffing against server API, exploitation of known CVEs |
| **Security researcher** | Responsible disclosure, reputation | High — hardware + RF + software skills | All levels | Full spectrum; likely to find vulnerabilities in RF protocol or boot chain |

## 3. Attack Surface Enumeration

### 3.1 Physical Attack Surface (Node)

| Attack Point | Description | Accessible To |
|-------------|-------------|--------------|
| Debug port (SWD) | 4-pin header on B-U585I-IOT02A  | Direct physical access |
| Flash extraction via SWD | Read application flash if RDP < Level 2 | Direct physical access |
| Node relocation | Move sensor to wrong space | Direct physical access |
| Enclosure tamper | Open device, probe internal signals | Direct physical access |
| Power rail manipulation | Glitching to bypass secure boot | Skilled attacker, direct access |
| I2C bus probing | Monitor SDA/SCL to extract sensor data | Skilled attacker, direct access |

### 3.2 RF Attack Surface (Zigbee 2.4 GHz)

| Attack Point | Description | Accessible To |
|-------------|-------------|--------------|
| Passive eavesdropping | Capture Zigbee frames with SDR | Anyone within ~300 m |
| Replay attack | Record and retransmit valid frames | Anyone within range |
| Rogue node join | Malicious device joins PAN | Anyone within range |
| Beacon spoofing | Advertise fake PAN to attract nodes | Anyone within range |
| RF jamming | Flood 2.4 GHz channel 15 with noise | Anyone with commodity RF hardware |
| Frame injection | Inject forged frames without NWK key | Anyone — rejected by AES-128-CCM |
| Channel scanning | Map network topology, count nodes | Anyone within range with SDR |

### 3.3 Network Attack Surface (Gateway → Server)

| Attack Point | Description | Accessible To |
|-------------|-------------|--------------|
| WiFi MITM | Rogue AP or ARP poisoning on LAN | LAN-adjacent attacker |
| TLS certificate spoofing | Present forged server cert if CA not pinned | LAN-adjacent attacker |
| TCP connection hijack | TCP RST injection | LAN-adjacent attacker |
| Server API attack | Malformed packets to server TCP port | Network-adjacent attacker |
| WiFi PSK theft | Extract WiFi password from gateway flash | Physical attacker |

### 3.4 Supply Chain Attack Surface

| Attack Point | Description |
|-------------|-------------|
| Firmware signing key compromise | GitHub Actions secret leak; CI/CD pipeline breach |
| Malicious pull request | Backdoor inserted in code review bypass |
| Compromised vendor library | Malicious update to VL53L5CX ULD, iis2mdctr, or X-CUBE-MXCHIP |
| Counterfeit hardware | Fake MCU/RF module during production |

## 4. STRIDE Analysis

STRIDE (Spoofing, Tampering, Repudiation, Information Disclosure, Denial of Service, Elevation of Privilege) applied to ParkSense data flows.

### 4.1 Node → Gateway (Zigbee)

| STRIDE Category | Threat | Control |
|----------------|--------|---------|
| **Spoofing** | Rogue node injects occupancy reports with a forged NODE_ID | Zigbee Trust Center Install Code auth; AES-128-CCM prevents unsigned frames |
| **Tampering** | Attacker flips OCCUPANCY byte in transit | AES-128-CCM MIC (4 bytes) detects modification; NWK layer discards |
| **Repudiation** | Node denies transmitting a report | Frame counter + timestamp + node address logged by gateway |
| **Information Disclosure** | Adversary reads occupancy state from RF | AES-128-CCM encrypts payload; only PAN members hold NWK key |
| **Denial of Service** | RF jamming prevents node → gateway communication | Zigbee channel agility (future); alert on missed heartbeats |
| **Elevation of Privilege** | Attacker uses Zigbee access to control gateway or server | Zigbee carries data only; no remote code execution path via Zigbee |

### 4.2 Gateway → Server (WiFi / TCP / TLS)

| STRIDE Category | Threat | Control |
|----------------|--------|---------|
| **Spoofing** | Rogue server accepts gateway connection (MITM) | Server X.509 certificate validated against embedded CA cert |
| **Tampering** | CPM packet modified in transit | TLS record integrity (HMAC-SHA256 in TLS 1.2) |
| **Repudiation** | Gateway denies forwarding a packet | Server timestamp + source IP logged |
| **Information Disclosure** | TCP stream intercepted | TLS 1.2 encryption |
| **Denial of Service** | TCP RST injection drops connection | Auto-reconnect within 5 s; server alerts on gateway silence |
| **Elevation of Privilege** | Attacker pivots from gateway WiFi to internal LAN | Gateway only opens TCP to server; no inbound connections; no shell |

### 4.3 Firmware Update (OTA)

| STRIDE Category | Threat | Control |
|----------------|--------|---------|
| **Spoofing** | Attacker delivers unsigned firmware image | ECDSA P-256 verification by bootloader; boot halts if invalid |
| **Tampering** | Image modified after signing | SHA-256 + CRC-32 in header; any bit flip detected |
| **Repudiation** | Cannot attribute which update was applied | Version field in image header; logged in fault record |
| **Information Disclosure** | Firmware binary reveals proprietary algorithms | Firmware is not encrypted (not secret); signing provides integrity only |
| **Denial of Service** | Attacker sends corrupt image repeatedly | Download timeout (30 min); failed verify → erase + continue running current |
| **Elevation of Privilege** | OTA delivers image with elevated permissions | Bootloader enforces minimum version; ECDSA prevents unauthorized images |

### 4.4 Physical Access to Node

| STRIDE Category | Threat | Control |
|----------------|--------|---------|
| **Spoofing** | Attacker replaces node with malicious device | Node has unique EUI-64 address burned in; gateway detects address mismatch |
| **Tampering** | Attacker writes alternate firmware via SWD | RDP Level 2 in production; ECDSA prevents running unsigned image |
| **Information Disclosure** | Flash readback of production firmware | RDP Level 2 disables flash readback entirely |
| **Denial of Service** | Physical destruction of node | Monitoring alerts on missing heartbeats |
| **Elevation of Privilege** | Direct SWD gives full MCU control | RDP Level 2 disables SWD permanently in production |

## 5. Attack Trees

### 5.1 Attack Tree: Inject False Occupancy Data

```
GOAL: Cause dashboard to show incorrect occupancy state
│
├── [A] Compromise a legitimate node
│     ├── [A1] Flash malicious firmware via SWD
│     │         Requires: Physical access + RDP not Level 2
│     │         Blocked by: RDP Level 2 (production)
│     │
│     └── [A2] Deliver malicious OTA update
│               Requires: Valid ECDSA private key
│               Blocked by: GitHub Actions secret + ECDSA verification
│
├── [B] Inject Zigbee frame as rogue node
│     ├── [B1] Join PAN without Install Code
│     │         Requires: Know Install Code OR default join = open
│     │         Blocked by: Trust Center requires Install Code enrollment
│     │
│     └── [B2] Replay a captured legitimate frame
│               Requires: Capture frame within range
│               Blocked by: Zigbee frame counter replay protection
│
├── [C] Compromise gateway and modify data in transit
│     ├── [C1] Gain shell on gateway
│     │         Requires: Remote code execution (no inbound ports exposed)
│     │         Blocked by: No listening ports; no remote shell service
│     │
│     └── [C2] Modify CPM packet in gateway RAM via debug port
│               Requires: SWD access to gateway
│               Blocked by: RDP Level 2 on gateway
│
└── [D] Compromise server database directly
      Not in firmware scope; server security is a separate concern
```

**Conclusion:** All firmware-reachable attack paths are blocked by enforced controls. The residual risk is supply chain (signing key theft) and physical destruction (out of scope for firmware).

### 5.2 Attack Tree: Extract Cryptographic Material

```
GOAL: Obtain Zigbee NWK key or WiFi PSK
│
├── [A] Read flash contents
│     Requires: SWD access + RDP < Level 2
│     Blocked by: RDP Level 2
│
├── [B] Probe SRAM3_S (Secure World) at runtime
│     Requires: Code execution in NS world attempting S-region access
│     Blocked by: TrustZone SAU; triggers SecureFault → reset
│
├── [C] Sniff Zigbee key distribution frame on air
│     Requires: Know Join Link Key (Install Code derived)
│     Blocked by: Install Code is per-device; NWK key encrypted with it
│
├── [D] Find key in firmware binary
│     Keys are not compiled into firmware binary; loaded from SRAM3_S at runtime
│     WiFi PSK is written to SRAM3_S at gateway provisioning
│     Blocked by: SRAM3_S inaccessible from NS world
│
└── [E] Supply chain: compromise CI/CD to embed key in binary
      Requires: GitHub Actions access + bypass code review
      Mitigated by: Secret scanning; branch protection; code review requirement
```

## 6. Risk Register

All threats scored using: **Risk = Likelihood × Impact** (1–5 scale each).

| ID | Threat | Likelihood | Impact | Risk | Residual (post-controls) |
|----|--------|-----------|--------|------|--------------------------|
| T-01 | Malicious firmware via SWD | 2 (needs physical; RDP2 in prod) | 5 (full device control) | 10 | 2 (RDP2 blocks; prototype risk only) |
| T-02 | Unsigned OTA image injection | 2 (needs ECDSA key) | 5 (full code control) | 10 | 2 (ECDSA blocks) |
| T-03 | Zigbee eavesdropping | 4 (SDR is cheap) | 1 (occupancy data is public-adjacent) | 4 | 2 (AES-128 encrypts) |
| T-04 | Zigbee replay attack | 3 | 3 (false state for ≤1 cycle) | 9 | 2 (frame counter) |
| T-05 | Rogue node join (no Install Code) | 3 | 3 (inject false data) | 9 | 2 (Trust Center enrollment) |
| T-06 | WiFi MITM (TLS) | 2 (needs LAN access) | 3 (data integrity) | 6 | 1 (cert validation) |
| T-07 | Physical node relocation | 4 (open lot) | 3 (wrong space reported) | 12 | 8 (heartbeat alert; no firmware control) |
| T-08 | RF jamming (DoS) | 3 | 3 (reduced availability) | 9 | 6 (no firmware mitigation in Phase 1) |
| T-09 | WiFi PSK exfiltration from binary | 2 | 4 (network access) | 8 | 1 (SRAM3_S + RDP2) |
| T-10 | SWD debug left open (dev build) | 3 (human error) | 5 | 15 | 4 (CI/CD build flag check) |
| T-11 | Compromised signing key (supply chain) | 1 (well-protected secret) | 5 | 5 | 2 (dual approval, secret scanning) |
| T-12 | Malicious vendor library update | 2 | 4 | 8 | 4 (vendored + pinned; no auto-update) |
| T-13 | Beacon spoofing (PAN hijack) | 2 | 3 (node joins wrong PAN) | 6 | 3 (Install Code; NWK key unique to PAN) |
| T-14 | Server API exploitation | 2 (network boundary) | 4 | 8 | Out of firmware scope |

**Accepted residual risks:**
- **T-07** (physical relocation): Mitigated by physical installation (bolted hardware, tamper-evident seal). Not addressable in firmware.
- **T-08** (RF jamming): Zigbee channel agility planned for Phase 2 (FR-SEC-002 backlog).

|----------------|---------------------|
| I1: Weak/Guessable Credentials | **Mitigated** — no user-facing passwords on device; WiFi PSK in SRAM3_S |

| I7: Insecure Data Transfer | **Mitigated** — Zigbee AES-128-CCM + TLS 1.2 |
| I8: Lack of Device Management | **Partial** — OTA update supported; no remote configuration interface in Phase 1 |
| I9: Insecure Default Settings | **Mitigated** — unique per-device Install Code; no universal defaults |
