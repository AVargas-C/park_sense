## 1. Purpose of This View

The Security Architecture view defines how ParkSense protects the confidentiality, integrity, and availability of the system. It identifies trust boundaries, documents the threat model, and specifies the security controls applied at each layer — from silicon-level hardware isolation to application-layer cryptography.

**Concerns addressed:**
- What are the system's trust boundaries?
- What are the primary threat categories?
- How does the device establish a root of trust at boot?
- How is firmware authenticity verified before and during OTA updates?
- How is wireless communication protected?
- How are cryptographic keys managed?
- What security controls does the server apply?

## 3. Threat Model

| Threat ID | Category | Description | Impacted Asset | Likelihood | Severity |
| --------- | -------- | ----------- | -------------- | ---------- | -------- |
| T-01 | Firmware Tampering | Adversary flashes malicious firmware via JTAG/SWD | Node firmware integrity | Low (physical access required) | Critical |
| T-02 | OTA Image Injection | Adversary delivers unsigned image over Zigbee OTA | Node firmware integrity | Medium | Critical |
| T-03 | Zigbee Eavesdropping | Passive sniff of occupancy packets on 2.4 GHz | Parking data confidentiality | High (RF is open) | Low (data not sensitive) |
| T-04 | Zigbee Replay Attack | Replay old occupancy report to forge state | Occupancy data integrity | Medium | Medium |
| T-05 | Rogue Node Join | Malicious device joins PAN and injects false reports | Occupancy data reliability | Medium | High |
| T-06 | WiFi MITM | ARP poisoning or rogue AP intercepts gateway uplink | Occupancy data integrity | Low (encrypted TLS) | Medium |
| T-07 | Physical Node Tamper | Attacker removes/relocates node hardware | Sensor coverage | Medium | Medium |
| T-08 | Denial of Service (RF) | Jamming 2.4 GHz channel | System availability | Medium | Medium |
| T-09 | Credential Exfiltration | WiFi PSK leaked from firmware binary | Gateway → network access | High (without protection) | High |
| T-10 | Debug Port Exposure | SWD left open in production firmware | Firmware confidentiality | Medium | High |

## 4. Security Controls

### 4.1 Hardware Root of Trust — STM32U585 TrustZone

The STM32U585AII6Q implements Arm TrustZone-M (Cortex-M33). The Security Attribution Unit (SAU) and Implementation-Defined Attribution Unit (IDAU) partition the address space:

| Region | Security | Contents |
| ------ | -------- | -------- |
| Flash sector 0 (bootloader) | Secure (S) | Bootloader + ECDSA verifier |
| Flash sector 1 (application) | Non-Secure (NS) | Application firmware |
| SRAM3 | Secure (S) | Private key material, session secrets |
| SRAM1, SRAM2 | Non-Secure (NS) | Application data, sensor buffers, packet queues |
| HW crypto (AES, PKA, SHA, OTFDEC) | Secure (S) | Accessible by bootloader and secure services |

The NS world cannot read or write S memory regions — enforced by hardware. The bootloader configures SAU before transferring control to the NS application.

**Mitigates:** T-01, T-02, T-09, T-10

### 4.2 Secure Boot Chain

```
                    POWER ON
                       │
                       ▼
            ┌──────────────────────┐
            │  ROM Boot (ST OTP)   │ ← Verifies bootloader using ST OTP hash
            └──────────────────────┘
                       │  SHA-256 hash match
                       ▼
            ┌──────────────────────┐
            │  ParkSense Bootloader│ ← Runs in Secure World
            │  (Flash sector 0)    │
            │                      │
            │  1. Read image header│ ← 256-byte header at start of application sector
            │  2. Verify CRC-32    │ ← Reject if mismatch
            │  3. Verify ECDSA P-256 signature over image SHA-256 digest
            │  4. Verify version ≥ stored minimum (anti-rollback)
            └──────────────────────┘
                       │  All checks pass
                       ▼
            ┌──────────────────────┐
            │  Application Firmware│ ← NS world; boots into main_node/main_gateway
            └──────────────────────┘
```

**Image header format (256 bytes):**

| Offset | Field | Size | Description |
| ------ | ----- | ---- | ----------- |
| 0x00 | MAGIC | 4 B | `0x50 0x4B 0x53 0x46` ("PKSF") |
| 0x04 | VERSION | 4 B | Semantic version (major.minor.patch.build) as uint32 |
| 0x08 | IMAGE_LEN | 4 B | Application image size in bytes |
| 0x0C | CRC32 | 4 B | CRC-32 over application image bytes |
| 0x10 | SHA256_DIGEST | 32 B | SHA-256 hash of application image |
| 0x30 | ECDSA_SIG | 64 B | ECDSA P-256 signature over SHA256_DIGEST |
| 0x70 | MIN_VERSION | 4 B | Minimum allowed version (anti-rollback) |
| 0x74 | RESERVED | 140 B | Zeroed; reserved for future fields |

**Signing process (CI/CD):**

```
Build → SHA-256(image) → ECDSA P-256 sign (private key from GitHub Actions secret)
     → Append 256-byte header → Flash artifact → OTA update
```

The private signing key is never stored in the repository. It is injected at build time via GitHub Actions secrets.

**Mitigates:** T-01, T-02

### 4.3 Zigbee Security

Zigbee 3.0 provides two layers of symmetric encryption using AES-128-CCM:

#### Network Layer (NWK)

| Parameter | Value |
| --------- | ----- |
| Algorithm | AES-128-CCM |
| Key type | Network Key (NWK key) — shared among all devices in the PAN |
| Key distribution | Trust Center (gateway) distributes NWK key on join, encrypted with Join Link Key |
| Frame counter | 32-bit per-device frame counter prevents replay attacks (T-04) |
| Key update | Trust Center can broadcast a new NWK key; devices accept only if counter increments |

#### Application Layer (APS)

| Parameter | Value |
| --------- | ----- |
| Algorithm | AES-128-CCM |
| Key type | Application Link Key — unique per (node, coordinator) pair |
| Scope | Optionally applied to unicast APS frames in addition to NWK encryption |

**Trust Center behavior (gateway):**

| Event | Action |
| ----- | ------ |
| New device requests join | Verify Install Code (if configured) or default join key |
| NWK key sent to new device | Encrypted with APS link key of joining device |
| Rogue join attempt (no valid link key) | Reject at APS authentication step |

**Mitigates:** T-03, T-04, T-05

### 4.4 WiFi and Server Communication Security

| Layer | Security Control |
| ----- | ---------------- |
| WiFi (gateway → AP) | WPA2-PSK minimum; PSK stored in TrustZone secure storage, not hardcoded in NS flash |
| TCP transport | TLS 1.2 minimum via mbedTLS (onboard EMW3080B) |
| Certificate validation | Server presents X.509 certificate; gateway validates against embedded CA certificate |
| Payload | CPM packet already CRC-16 verified; server re-verifies before writing to database |

**Mitigates:** T-06, T-09

---

### 4.5 SWD/JTAG Debug Port Lockout

In production firmware builds, the STM32U585 RDP (Readout Protection) level is set to **RDP Level 2**:

| RDP Level | Effect |
| --------- | ------ |
| Level 0 | Debug access fully open (development only) |
| Level 1 | Debug access disabled; flash readback blocked; mass-erase allowed |
| Level 2 | Permanent — debug port disabled, mass-erase disabled. Irreversible. |

> **Warning:** RDP Level 2 is irreversible. Only apply to production-programmed devices.

Development and prototype builds use RDP Level 0 to allow JTAG debugging. The CI/CD production build step sets RDP Level 2 as a post-flash step using STM32CubeProgrammer.

**Mitigates:** T-10

### 4.6 OTA Firmware Update Security

OTA updates are distributed over Zigbee using the Zigbee OTA Upgrade cluster (ZCL cluster 0x0019). Security properties:

| Property | Implementation |
| -------- | -------------- |
| Authenticity | ECDSA P-256 signature verified by bootloader before any boot |
| Integrity | CRC-32 + SHA-256 in image header; bootloader checks both |
| Confidentiality | OTA image is not encrypted (signature is sufficient; binary is not secret) |
| Anti-rollback | Bootloader rejects images with VERSION < MIN_VERSION stored in OTP |
| Delivery | Zigbee NWK encryption wraps OTA cluster frames |

**OTA update flow:**

```
CI/CD → Signed firmware artifact
  │
  ▼
Gateway receives OTA image file
  │
  ├── Saves to designated OTA flash sector
  ├── Advertises OTA upgrade availability on Zigbee OTA cluster
  │
  ▼
Node polls gateway (Query Next Image Request)
  ├── Gateway responds: image available (version, size)
  ├── Node downloads image block-by-block (Image Block Requests)
  ├── Node writes to OTA download sector in flash
  └── Node signals bootloader to validate and swap → reboot
```

```
                │  Trust Root                       │
                │  ECDSA P-256 Public Key           │
                │  Embedded in bootloader at build  │
                └──────────────────────────┬────────┘
                                           │ verifies
                            ┌──────────────▼────────────┐
                            │ Firmware Image             │

          Zigbee Key Hierarchy:
          ┌─────────────────────────────────────────┐
          │ Global Install Code (per-device)         │ ← Pre-programmed at factory
          │   └─► Join Link Key (derived from IC)   │
          │         └─► Encrypts NWK Key delivery   │
          │               └─► NWK Key (AES-128)     │
          │                     └─► Encrypts frames │
          └─────────────────────────────────────────┘

          WiFi Credential:
          ┌─────────────────────────────────────────┐
          │ WiFi PSK                                 │
          │   Stored in: TrustZone SRAM3 (S) at boot│
          │   NOT in: NS flash, source code, binary  │
          └─────────────────────────────────────────┘
```

## 6. Physical Security Considerations

| Threat | Control |
| ------ | ------- |
| T-07: Node relocation | GPS is out of scope for Phase 1. Nodes are physically bolted to pavement/pole. Watchdog alert if node goes silent. |
| T-07: Node tamper | Enclosure with tamper-evident seal (hardware design scope). Firmware: no sensitive data in NS flash. |
| T-08: RF jamming | Zigbee channel agility — coordinator can perform channel scan and request network-wide channel change. Not implemented in Phase 1; logged as FR-SEC-002 in backlog. |
