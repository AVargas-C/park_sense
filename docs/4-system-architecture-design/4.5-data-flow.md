## 1. Purpose of This View

The Data Flow view traces the journey of parking occupancy information — from raw sensor measurements on the IoT node, through the Zigbee network, through the gateway, across WiFi to the server, and finally to the operator dashboard. It defines what data looks like at each stage of processing and what transformations are applied.

**Concerns addressed:**
- Where does data originate?
- What transformations does it undergo?
- What does the data look like at each stage?
- What are the latency and size budgets at each stage?
- What triggers data transmission?

## 3. Stage-by-Stage Data Description

### Stage 0: Physical World → Sensor

| Sensor | Raw Output | Rate | Size |
| ------ | ---------- | ---- | ---- |
| VL53L5CXV0GC/1 | 8×8 array of uint16_t distances (mm) | Up to 15 fps (configurable) | 128 × 2 B = 256 B per frame |
| IIS2MDCTR | int16_t[3] magnetic field (mGauss) for X, Y, Z | Up to 100 Hz (configurable) | 6 B per reading |

**Note on sampling rate:** The application configures sensors at a low rate to save power. Typical operating configuration:
- VL53L5CX: 1 sample/s (motion-triggered wake → sample → sleep)
- IIS2MDCTR: 10 Hz during occupancy-change detection window; 1 Hz in steady state

---

### Stage 1: Raw Sensor → PDM Intermediate

**`pdm_tof` processing:**

```
Input:  VL53L5CX_ResultsData (256 B — 64 distances)
Steps:
  1. Select minimum distance across all 64 zones          → uint16_t min_dist_mm
  2. Discard zones with sigma > threshold (invalid zones)
  3. Compare min_dist_mm to VEHICLE_PRESENCE_THRESHOLD    → bool tof_occupied
Output: tof_status_t {TOF_FREE, TOF_OCCUPIED, TOF_INVALID}  (1 byte)
```

**`pdm_mag` processing:**

```
Input:  mag_data_t {x_mGauss, y_mGauss, z_mGauss}
Steps:
  1. Compute magnitude: |B| = sqrt(x² + y² + z²)         → float magnitude_mGauss
  2. Compute deltas from calibration baseline: ΔB = |B| - |B_baseline|
  3. Compare ΔB to MAGNETIC_DISTURBANCE_THRESHOLD         → bool mag_disturbed
Output: mag_status_t {MAG_CLEAR, MAG_DISTURBED}             (1 byte)
```

Calibration baseline is captured during parking space provisioning (empty space reading stored in flash).

---

### Stage 2: PDM Intermediate → PDM Decision (Occupancy State)

**`pdm_fsm` processing:**

```
Inputs:  tof_status_t, mag_status_t
Logic:   OCCUPIED if (tof_occupied OR mag_disturbed) for N consecutive samples
         FREE if (tof_free AND mag_clear) for M consecutive samples
         Hysteresis: N = 3 (enter OCCUPIED), M = 5 (return to FREE)
States:  FREE, DEBOUNCE, OCCUPIED

Output:  pdm_state_t {PDM_FREE, PDM_OCCUPIED, PDM_ERROR}   (1 byte)
         bool state_changed                                  (1 byte)
```

**Decision trigger rules:**
- `CPM_SendReport()` is called **only when `state_changed = true`** → minimizes unnecessary transmissions
- Heartbeat report is sent every `HEARTBEAT_INTERVAL_S` (default: 60 s) regardless of state change → allows gateway to detect dead nodes

---

### Stage 3: PDM Decision → CPM Packet

**CPM serialization:**

```
Input:  pdm_state_t, node_id (from BSP config), space_id (from BSP config)

Output: cpm_packet_t (12 bytes)

 ┌──────────────┬───────────────┬────────────────┬──────────────┬──────────────────┬──────────────┐
 │ MSG_TYPE (1B)│ NODE_ID  (2B) │ SPACE_ID  (2B) │ OCCUPANCY(1B)│ TIMESTAMP   (4B) │ CRC16   (2B) │
 └──────────────┴───────────────┴────────────────┴──────────────┴──────────────────┴──────────────┘
  Total: 12 bytes
```

**CRC calculation:** CRC-16/CCITT-FALSE polynomial `0x1021`, initial value `0xFFFF`, applied over bytes 0–9.

**TIMESTAMP field:** The IoT node does not have a real-time clock; the TIMESTAMP field is set to `0x00000000` and the gateway stamps the received frame with its local time (synchronized via SNTP).

---

### Stage 4: CPM Packet → Zigbee Frame (Node RF TX)

```
Input:  cpm_packet_t (12 bytes)

Steps:
  1. AES-128-CCM encrypt payload (Zigbee NWK layer, automatic via stack)
  2. Build IEEE 802.15.4 MAC frame:
       └── Frame Control | Seq Num | Dest PAN + Addr | Src Addr | Security Header | APS Header | CPM Payload | FCS
  3. Transmit via STM32WB Zigbee stack MAC

Output: IEEE 802.15.4 frame (~50–80 bytes over the air, including headers + security overhead)
Max PHY payload (MAC SDU): 127 bytes
```

**Duty cycle:** A single transmission event lasts approximately 2 ms at 250 kbps. Zigbee end device transmits, then polls for MAC ACK within 864 µs. If ACK is received, the node returns to sleep.

---

### Stage 5: Zigbee Frame → Gateway CPM Receive

```
Input:  IEEE 802.15.4 frame (received by STM32WB5MMG Zigbee stack)

Steps:
  1. Zigbee stack handles: FCS check, replay counter check, AES-128-CCM decryption
  2. APS layer strips headers → delivers CPM payload (12 bytes) to application
  3. CPM_ReceiveReport() parses the payload
  4. Verify CRC-16 over bytes 0–9 → reject if mismatch
  5. Gateway appends SNTP timestamp to TIMESTAMP field (if node field = 0)
  6. Route to CPM forwarding queue

Output: cpm_packet_t (12 bytes, decrypted, CRC verified, timestamp filled)
```

---

### Stage 6: CPM Packet → TCP/IP Frame (Gateway WiFi TX)

```
Input:  cpm_packet_t (12 bytes)

Steps:
  1. Wrap CPM payload in a TCP/IP datagram:
       └── [Length (2B)] [CPM Packet (12B)]   → 14 bytes application payload
  2. TLS 1.2 encryption via mbedTLS (onboard EMW3080B)
  3. TCP transmission to server endpoint

Output: TCP segment (14 bytes payload) over TLS 1.2 connection
```

**Alternative protocol:** If memory constraints allow, the gateway may use an MQTT PUBLISH message instead of raw TCP — specified in Firmware Architecture (Phase 2 doc, not yet written).

---

### Stage 7: Server TCP Receive → Database + WebSocket

```cpp
Input: TCP frame (14 bytes), parsed to cpm_packet_t

Steps:
  1. Validate CRC-16 (second check at server boundary)
  2. Look up space_id in database → update occupancy record
  3. Record timestamp (server wall clock used if node timestamp = 0)
  4. Persist to database: (space_id, occupancy, timestamp, node_id)
  5. Broadcast WebSocket event to all connected dashboard clients

Output A: Database row update
Output B: JSON WebSocket push event
```

---

### Stage 8: WebSocket Event → Dashboard

```json
{
  "event": "OCCUPANCY_UPDATE",
  "node_id": 42,
  "space_id": "B-14",
  "occupancy": "occupied",
  "timestamp": 1743638400,
  "previous_state": "free",
  "duration_free_s": 1800
}
```

The dashboard renders the space map and updates the color of parking space `B-14` from green (free) to red (occupied) within the WebSocket event processing handler.

## 4. Heartbeat Flow

Heartbeat reports use `MSG_TYPE = 0x02` and `OCCUPANCY = 0xFF` (no change). The gateway and server use heartbeats to:

1. Confirm node is alive
2. Update the server-side "last seen" timestamp
3. Trigger an alert if a node is not heard within `HEARTBEAT_INTERVAL_S × 3` = 180 s

## 5. Data Size Summary

| Stage | Data | Size |
| ----- | ---- | ---- |
| VL53L5CX raw frame | 8×8 uint16_t distances | 256 B |
| IIS2MDCTR reading | int16_t[3] | 6 B |
| PDM outputs | 2× status enums | 2 B |
| CPM packet | Binary, fixed-format | 12 B |
| Zigbee MAC frame | CPM + all headers | ~50–80 B |
| TCP payload (gateway → server) | Length prefix + CPM | 14 B |
| WebSocket JSON event | JSON string | ~200 B |

## 6. Latency Budget

| Segment | Typical Latency | Max Allowed |
| ------- | --------------- | ----------- |
| Sensor read (ToF + Mag) | 50 ms | 200 ms |
| PDM fusion + decision | < 1 ms | 5 ms |
| CPM serialize | < 1 ms | 1 ms |
| Zigbee TX + MAC ACK | 5–15 ms | 50 ms |
| Gateway CPM deserialize + enqueue | < 1 ms | 5 ms |
| Gateway WiFi TCP TX | 20–100 ms | 500 ms |
| Server process + DB write | 5–50 ms | 200 ms |
| WebSocket push to dashboard | 1–10 ms | 100 ms |
| **Total end-to-end** | **~80–180 ms** | **< 2 s** |

> System requirement SYS-P-001 specifies occupancy updates ≤ 2 s end-to-end. Budget analysis confirms compliance.

## 7. Error Data Flow

If a sensor read fails (I2C error, timeout), the PDM places the occupancy output into `PDM_ERROR` state. The CPM sends an error report (`OCCUPANCY = 0xFF`, `MSG_TYPE = 0x01`) to the gateway. The server records the fault against the node ID and the dashboard marks the space with a yellow "unknown" indicator.

```
Sensor fault
  └── TOF_GetResults() → TOF_ERR_TIMEOUT
        └── pdm_fsm → PDM_ERROR state
              └── CPM_SendReport(PDM_ERROR)
                    └── MSG_TYPE=0x01, OCCUPANCY=0xFF (error)
                          └── Server → space status = UNKNOWN → dashboard yellow
```
