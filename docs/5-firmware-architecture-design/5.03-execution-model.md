## 1. Purpose

This view describes *when* code executes. It defines the super-loop structure for both the IoT Node and Gateway, catalogs every interrupt, specifies the cooperative scheduling mechanism, and proves that the timing budget meets the SyRS performance requirements.

---

## 2. Execution Strategy

ParkSense uses a **cooperative bare-metal super loop** — no RTOS kernel, no preemptive scheduling, no context switching (see AD-002).

| Property | Value |
|----------|-------|
| Kernel | None (bare metal) |
| Scheduling | Cooperative — tasks run to completion, yield voluntarily |
| Preemption | ISRs only; all ISRs are short (< 10 µs) and set volatile flags |
| Context switches | None |
| Stack | Single stack per target (no task stacks) |
| Timing reference | SysTick (1 ms tick) + RTC alarm (30 s wake) |

**Why no RTOS:** The workload is simple and sequential — read sensors, decide, transmit, sleep. An RTOS adds ~8–15 KB flash, per-task stacks (4 KB each), and non-deterministic scheduling. The cooperative model is more predictable, lower power, and easier to verify.

## 3. IoT Node — Super Loop

### 3.1 Complete Execution Sequence

```
POWER ON
  │
  ▼
Reset_Handler (startup_stm32u585xx.s)
  │  zero .bss, copy .data from flash → SRAM
  ▼
SystemInit()
  │  Configure flash wait states, MSI → PLL → 160 MHz
  ▼
main() [main_node.c]
  │
  ├── BSP_Init()
  │     Clock: PLL → 4 MHz (downscale after boot)
  │     GPIO: configure I2C1 pins, LPn, EXTI, IWDG
  │     RTC: init calendar, no alarm yet
  │     IWDG: start (4 s timeout)
  │
  ├── PDM_Init()
  │     Load calibration from CONFIG_FLASH → SRAM2
  │     Init FSM to FREE state
  │
  ├── CPM_Init()
  │     Init TX/RX queues (zero fill)
  │     Load node_id, space_id from CONFIG_FLASH
  │
  ├── RF_Init()
  │     Configure STM32WB5 as Zigbee Sleepy End Device
  │
  ├── RF_JoinNetwork(timeout_ms=30000)
  │     Scan → Associate → Receive NWK key
  │     On failure: log fault, sleep 60 s, retry
  │
  └── SUPER LOOP ─────────────────────────────────────
        │
        ▼
  ┌──► [1] IWDG_Refresh()                         ← feed watchdog
  │         │
  │         ▼
  │    [2] TOF_Wake() + MAG_Wake()                 ← power up sensors
  │         │   LPn → HIGH, MAG → continuous mode
  │         ▼
  │    [3] TOF_Start()                             ← trigger one ranging cycle
  │         │   wait VL53L5CX data-ready (polled, ~33 ms)
  │         ▼
  │    [4] TOF_GetResults(&tof_data)               ← read 8×8 I2C burst
  │         │
  │         ▼
  │    [5] MAG_Read(&mag_data)                     ← 6-byte I2C read (~1 ms)
  │         │
  │         ▼
  │    [6] PDM_Update(tof_data, mag_data)
  │         │   pdm_tof → min-distance threshold
  │         │   pdm_mag → delta magnitude threshold
  │         │   pdm_fsm → occupancy state + debounce
  │         ▼
  │    [7] if (PDM_StateChanged())
  │         │   CPM_SendReport(PDM_GetOccupancy())
  │         │     Serialize → CRC → RF_Send() → wait ACK
  │         │     On NACK/timeout → cpm_retry (up to 3×)
  │         │
  │         ▼
  │    [8] if (heartbeat_timer_expired())
  │         │   CPM_Heartbeat()
  │         │     Serialize heartbeat → RF_Send()
  │         │
  │         ▼
  │    [9] TOF_Sleep() + MAG_Sleep()               ← power down sensors
  │         │   LPn → LOW, MAG → power-down register
  │         ▼
  │   [10] RF_Sleep()                              ← STM32WB radio → deep sleep
  │         │
  │         ▼
  │   [11] PWR_ConfigureWakeRTC(SLEEP_INTERVAL_S)  ← set RTC alarm = now + 30 s
  │         │
  │         ▼
  │   [12] PWR_EnterStop2()                        ← CPU + peripherals off, SRAM2 retained
  │         │
  │         │   ═══════ SLEEPING (29.93 s) ═══════
  │         │
  │         ▼
  │   [13] RTC_Alarm_IRQHandler fires              ← wake from Stop 2
  │         │   Restore MSI → PLL → 4 MHz
  │         │   Re-enable peripheral clocks
  │         │
  └────────┘
```

### 3.2 Timing Budget per Cycle

| Step | Duration | Notes |
|------|----------|-------|
| [1] IWDG refresh | < 0.1 ms | Single register write |
| [2] Sensor wake | ~5 ms | LPn rise time + VL53L5CX boot (typ. 1.3 ms) |
| [3–4] ToF ranging + read | ~35 ms | Single-shot ranging at 30 Hz integration; I2C burst 256 B @ 400 kHz |
| [5] MAG read | ~1 ms | 6-byte I2C read |
| [6] PDM compute | ~2 ms | Integer arithmetic, no floating point in steady state |
| [7] Zigbee TX + ACK | ~15 ms | Includes RF wake + frame build + airtime + MAC ACK wait |
| [8] Heartbeat | 0 or ~15 ms | Only every other cycle (60 s interval) |
| [9] Sensor sleep | < 1 ms | GPIO write + I2C register write |
| [10] RF sleep | < 1 ms | IPCC command |
| [11] RTC alarm setup | < 0.1 ms | |
| [12–13] Stop 2 sleep | 29.93 s | Dominant |
| **Active total** | **~60 ms** | Well within SYS-P-001 (≤ 1 s) |

**Compliance:** Active phase ≈ 60 ms << 1 s limit (SYS-P-001). ✓

## 4. Gateway — Super Loop

### 4.1 Complete Execution Sequence

```
POWER ON
  │
  ▼
Reset_Handler → SystemInit() → main() [main_gateway.c]
  │
  ├── BSP_Init()
  │     Clock: PLL → 4 MHz
  │     GPIO: SPI2 + WiFi IRQ + IPCC
  │     IWDG: start (4 s timeout)
  │
  ├── RF_CoordinatorInit()
  │     Configure STM32WB5 as Zigbee Coordinator
  │     RF_PermitJoining(0xFF)  ← always open during dev; time-limited in production
  │
  ├── WiFi_Connect(ssid, password)
  │     SPI2 init → EMW3080B boot → scan → associate → DHCP
  │     On failure: retry with exponential backoff (1 s, 2 s, 4 s, max 30 s)
  │
  ├── WiFi_TCPConnect(server_host, server_port)
  │     DNS resolve → TCP connect → TLS handshake
  │
  ├── CPM_Init()
  │     Init TX/RX queues
  │
  └── SUPER LOOP ─────────────────────────────────────
        │
        ▼
  ┌──► [1] IWDG_Refresh()
  │         │
  │         ▼
  │    [2] RF_CoordinatorReceive(&rx_frame)        ← non-blocking poll
  │         │   Returns RF_OK if frame available, RF_ERR_EMPTY if none
  │         ▼
  │    [3] if (frame_available)
  │         │   CPM_Deserialize(rx_frame.data, &pkt)
  │         │   Verify CRC-16
  │         │   Stamp timestamp (SNTP or RTC)
  │         │   CPM_ForwardToServer(&pkt)
  │         │     WiFi_TCPSend(serialized_pkt, 14)
  │         │
  │         ▼
  │    [4] WiFi_TCPReceive(rx_buf, &rx_len)         ← non-blocking poll
  │         │   Handle server commands (future: OTA trigger, config push)
  │         ▼
  │    [5] if (wifi_keepalive_timer_expired())
  │         │   WiFi_TCPSend(keepalive_ping, 4)     ← TCP keepalive
  │         ▼
  │    [6] if (wifi_reconnect_needed())
  │         │   WiFi_TCPDisconnect()
  │         │   WiFi_Connect() → WiFi_TCPConnect()  ← reconnect
  │         ▼
  │    [7] __WFI()                                  ← wait for interrupt (saves power)
  │         │   Wakes on: Zigbee RX (IPCC IRQ), WiFi data-ready (EXTI), SysTick
  │         │
  └────────┘
```

### 4.2 Gateway Timing

The gateway has no strict cycle period. It is event-driven:
- Zigbee RX → process → WiFi TX → return to WFI
- Typical event processing: < 50 ms per packet
- SYS-P-006 requires forwarding within 500 ms — comfortably met

## 5. Interrupt Service Routine (ISR) Catalog

### 5.1 Node ISRs

| IRQ | Vector | Priority | Duration | Action |
|-----|--------|----------|----------|--------|
| `Reset_Handler` | 1 | — | N/A | Startup: `.bss` zero, `.data` copy, `SystemInit`, `main` |
| `NMI_Handler` | 2 | −2 (fixed) | < 1 µs | Log fault, trigger IWDG reset |
| `HardFault_Handler` | 3 | −1 (fixed) | < 10 µs | Log fault record to SRAM2, trigger IWDG reset |
| `MemManage_Handler` | 4 | 0 | < 10 µs | Stack overflow detection (MPU guard), log + reset |
| `BusFault_Handler` | 5 | 1 | < 10 µs | Log + reset |
| `UsageFault_Handler` | 6 | 2 | < 10 µs | Log + reset |
| `SysTick_Handler` | 15 | 15 (lowest) | < 1 µs | Increment `HAL_TickCount` (1 ms tick); increment app scheduler timers |
| `RTC_Alarm_IRQHandler` | 41 | 5 | < 5 µs | Set `g_rtc_alarm_flag = 1`; wake from Stop 2 |
| `I2C1_EV_IRQHandler` | 55 | 8 | < 5 µs | Used only in interrupt-mode I2C (optional; polled mode is default) |
| `IWDG_IRQHandler` | — | — | N/A | No ISR — IWDG timeout triggers hardware reset directly |

### 5.2 Gateway ISRs (additional)

| IRQ | Vector | Priority | Duration | Action |
|-----|--------|----------|----------|--------|
| `EXTI_WiFi_IRQ` | Config | 6 | < 5 µs | Set `g_wifi_data_ready = 1` (EMW3080B data-ready IRQ pin) |
| `IPCC_C1_RX_IRQHandler` | Config | 4 | < 5 µs | Set `g_zigbee_rx_pending = 1` (STM32WB5 mailbox notification) |

### 5.3 ISR Design Rules

1. **ISRs never call blocking functions.** No `HAL_Delay()`, no `HAL_I2C_Mem_Read()` inside ISRs.
2. **ISRs set volatile flags only.** The super loop polls these flags and processes data in main context.
3. **ISR total execution < 10 µs.** Ensures zero impact on timing budget.
4. **Priority scheme:** Fault handlers > RTC alarm > IPCC/WiFi events > SysTick. SysTick is lowest priority to avoid delaying critical interrupts.

## 6. Cooperative Scheduler

The application layer uses a lightweight tick-driven scheduler to manage periodic tasks without an RTOS.

### 6.1 Scheduler Design

```c
typedef void (*task_fn_t)(void);

typedef struct {
    task_fn_t   fn;           /* function pointer */
    uint32_t    period_ms;    /* execution period */
    uint32_t    last_tick;    /* last execution tick (SysTick counter) */
    bool        enabled;      /* runtime enable/disable */
} sched_task_t;
```

The scheduler is a simple array of tasks, scanned each iteration of the super loop:

```c
void SCHED_Run(void)
{
    uint32_t now = HAL_GetTick();
    for (uint32_t i = 0; i < SCHED_TASK_COUNT; i++) {
        if (g_tasks[i].enabled &&
            (now - g_tasks[i].last_tick >= g_tasks[i].period_ms)) {
            g_tasks[i].fn();
            g_tasks[i].last_tick = now;
        }
    }
}
```

### 6.2 Registered Tasks

#### Node Tasks

| Task | Period | Description |
|------|--------|-------------|
| `SCHED_MeasureCycle` | 30,000 ms | Main sensing cycle (steps 2–9 of super loop) |
| `SCHED_Heartbeat` | 60,000 ms | Transmit heartbeat packet via CPM |
| `SCHED_IWDGRefresh` | 2,000 ms | Feed independent watchdog |

> On the node, `SCHED_MeasureCycle` is effectively the entire active-phase workload. Between calls, the node is in Stop 2 — so the scheduler only runs once per wake.

#### Gateway Tasks

| Task | Period | Description |
|------|--------|-------------|
| `SCHED_ProcessZigbee` | 0 (every loop) | Poll `g_zigbee_rx_pending`, process packets |
| `SCHED_ProcessWifi` | 0 (every loop) | Poll `g_wifi_data_ready`, receive server commands |
| `SCHED_WifiKeepalive` | 30,000 ms | Send TCP keepalive to server |
| `SCHED_WifiReconnect` | 5,000 ms | Check WiFi connection state, reconnect if lost |
| `SCHED_IWDGRefresh` | 2,000 ms | Feed independent watchdog |

## 7. Watchdog Strategy

### 7.1 Independent Watchdog (IWDG)

| Parameter | Value |
|-----------|-------|
| Clock source | LSI (32 kHz) — independent of system clock |
| Prescaler | 64 |
| Reload value | 2000 |
| Timeout | (2000 × 64) / 32,000 = **4.0 seconds** |
| Feed interval | Every 2 s via `SCHED_IWDGRefresh` |

**Behavior:**
- If the super loop hangs (infinite loop, I2C bus lockup, stack overflow), the watchdog is not fed within 4 s → hardware reset.
- The IWDG continues running during Stop 2 if clocked by LSI. **Caveat:** On STM32U5, the IWDG can be frozen in Stop modes via `DBGMCU_APB1FZR1`. For Stop 2, the IWDG is **stopped** (does not run during sleep). This is correct behavior — the node is not executing code during sleep, so a watchdog reset is undesirable.

### 7.2 Stop 2 Wake Safety

During Stop 2 sleep, the IWDG is frozen. The only wake source is the RTC alarm. If the RTC fails to fire (hardware fault), the node remains in Stop 2 indefinitely — a **dead node** scenario.

**Mitigation:** The backup SRAM stores a `last_wake_epoch` counter. If the server detects that a node has not sent a heartbeat in 3× the heartbeat interval (180 s), it flags the node as unresponsive on the dashboard. Field maintenance then investigates.

> This is a hardware-level risk accepted in the current design. A future revision may use LPTIM as a secondary wake source.

## 8. Clock Configuration

### 8.1 Boot Clock Path

```
Power-on → HSI16 (16 MHz, default)
  └── SystemInit()
        └── MSI → PLL → 160 MHz (used during: boot, ECDSA verify, OTA)
              └── BSP_Init()
                    └── PLL → 4 MHz (runtime operating frequency)
```

### 8.2 Clock Sources Active Per Mode

| Mode | SYSCLK | HSI16 | MSI | LSI | LSE | RTC |
|------|--------|-------|-----|-----|-----|-----|
| Run (4 MHz) | MSI 4 MHz | Off | On | On (IWDG) | Off | On |
| Run (160 MHz) | PLL → 160 MHz | Off | On (PLL source) | On | Off | On |
| Stop 2 | Off | Off | Off | On (IWDG, frozen if DBGMCU set) | Off | On (wake source) |

### 8.3 Wake-from-Stop 2 Clock Restoration

On RTC alarm interrupt, the CPU wakes with MSI at 4 MHz (default after Stop 2 exit). The firmware:

1. Re-enables peripheral clocks (`__HAL_RCC_I2C1_CLK_ENABLE()`, etc.)
2. Does **not** re-init PLL to 160 MHz — 4 MHz is sufficient for sensing + transmitting
3. Re-configures GPIO pins that were reset during Stop 2 (if GPIO retention is not enabled)

This avoids PLL lock time (~200 µs) on every wake cycle.
