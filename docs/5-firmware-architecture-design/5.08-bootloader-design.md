## 1. Purpose

This document specifies the design of the ParkSense secure bootloader: its security properties, complete boot decision flow, image validation sequence, OTA slot management, anti-rollback enforcement, fault handling, and hardware security configuration. The bootloader is the **root of trust** for the entire system — every security guarantee depends on it executing correctly before any application code runs.

---

## 2. Fundamental Properties

### 2.1 Immutability

The bootloader is **permanently locked** and cannot be updated in the field. This is a deliberate design decision: a bootloader that can update itself provides no trustworthy root of trust.

| Mechanism | Configuration | Effect |
|-----------|--------------|--------|
| **SECWM1** option byte | `STRT=0, END=15` (Bank 1 pages 0–15) | Bootloader flash (including veneer page 15) is Secure — NS code cannot read or execute it |
| **WRP1** option byte | `WRP1A_STRT=0, WRP1A_END=15` | Pages 0–15 are write-protected — neither Secure nor NS code can erase or reprogram them |
| **HDP1** option byte | `HDPSTRT=0, HDPEND=15` | Pages 0–15 are hidden — SWD debugger and NS bus matrix cannot read-out contents |
| **RDP1** (production) | Option byte `RDP=0xBB` | Disables all debug access to NS world; regression to RDP0 requires full chip erase (destroys all keys and flash) |

To replace the bootloader: RDP regression → full chip erase → re-flash bootloader → re-provision Secure Config → re-flash application. This is the only path and requires physical access.

### 2.2 Non-Updatability Guarantee

Because WRP1 locks pages 0–15 unconditionally (including from Secure-world flash programming calls), no code path — not OTA, not a Secure veneer, not a debug probe — can modify bootloader flash after provisioning. The OTA system downloads only into the **App B slot** (`0x08100000`) and the bootloader validates it on the next power cycle.

## 3. Memory Layout (Bootloader-Relevant Regions)

Addresses and sizes are authoritative from [[5.02-memory-map]]. Reproduced here for reference.

| Region | Address | Size | Security | Role in boot |
|--------|---------|------|----------|-------------|
| Bootloader code | `0x08000000` | 120 KB | **SECURE + WRP + HDP** | Executed by CPU at reset |
| NSC Veneer | `0x0801E000` | 8 KB | **NSC + WRP** | NS→Secure call gates (not used during boot itself) |
| App A (primary) | `0x08020000` | 896 KB | NS | Normal boot slot |
| App B (OTA staging) | `0x08100000` | 896 KB | NS | OTA candidate slot |
| Secure Config | `0x081E0000` | 8 KB | **SECURE + SECWM2** | ECDSA public key, min version floor |
| OTA Control | `0x081E2000` | 8 KB | NS | Swap directive, anti-rollback version |
| Fault Log | `0x081FE000` | 8 KB | NS | 4 KB ring buffer of `fault_record_t` |

### 3.1 Application Image Layout Within a Slot

Each slot's first 256 bytes are occupied by the `ps_image_header_t` struct. The application vector table (and therefore first executable instruction) begins at **slot_base + 0x100**.

```
slot_base + 0x000  ┌──────────────────────────────┐
                   │  ps_image_header_t  (256 B)   │
slot_base + 0x100  ├──────────────────────────────┤  ← VTOR_NS set here
                   │  NS vector table              │
                   │  Application code             │
                   │  Application data             │
slot_base + N      └──────────────────────────────┘
```

The linker script for Node and Gateway places `.vectors` at `FLASH_APP_A + 0x100` (App A) or `FLASH_APP_B + 0x100` (App B).

## 5. Complete Boot Flow

### 5.1 Top-Level Sequence

```
                     ┌─────────────┐
            Reset    │             │
          ──────────►│  CPU starts │  Vector table at 0x08000000 (SECURE)
                     │  in Secure  │  SP = bootloader stack top
                     │  world      │  PC = ResetHandler (btl_main.c)
                     └──────┬──────┘
                            │
                            ▼
                    ┌───────────────┐
                    │ [1] Secure    │  BSP_Secure_Init():
                    │     HW Init   │  Flash latency → PLL → peripheral clocks
                    └──────┬────────┘  → SECWM verify → SAU → GTZC → Secure MPU
                           │           → LSE/RTC → IWDG start
                           │
                           ▼
                    ┌───────────────┐
                    │ [2] SECWM     │  Read FLASH_SECWM1R1 + FLASH_SECWM2R1
                    │   Integrity   │  Compare to expected values
                    │   Check       │
                    └──────┬────────┘
                   PASS    │     FAIL
                           │  ───────► fault_log_write(FAULT_SECWM_TAMPER)
                           │           NVIC_SystemReset()
                           ▼
                    ┌───────────────┐
                    │ [3] Read OTA  │  Read ota_control_t at 0x081E2000
                    │   Control     │  Validate struct CRC-32
                    └──────┬────────┘
                           │
              swap_magic == 0x5AA5AA5A?
                     YES   │    NO
              ─────────────┤    └──► candidate = App A (0x08020000)
              candidate =  │
              App B         │
              (0x08100000)  │
                           ▼
                    ┌───────────────┐
                    │ [4] Validate  │  (see §5.2 for full detail)
                    │   Candidate   │
                    │   Image       │
                    └──────┬────────┘
                   PASS    │     FAIL
                           │  ───────► (see §5.3 Failure Handling)
                           ▼
                    ┌───────────────┐
                    │ [5] Commit    │  Only if candidate was App B:
                    │   OTA Swap    │  Write ota_ctrl.committed = 0x01
                    │  (if App B)   │  Clear ota_ctrl.swap_magic = 0x00000000
                    └──────┬────────┘  Update ota_ctrl.min_version if higher
                           │
                           ▼
                    ┌───────────────┐
                    │ [6] Configure │  Write NS MPU regions via MPU_NS alias
                    │   NS MPU      │  App A: RX | App B: RW+XN | stack guard
                    └──────┬────────┘  SRAM1/2/4 | NS peripherals | bg deny-all
                           │
                           ▼
                    ┌───────────────┐
                    │ [7] Set       │  SCB_NS->VTOR = candidate + 0x100
                    │   VTOR_NS     │  (past 256-byte header)
                    └──────┬────────┘
                           │
                           ▼
                    ┌───────────────┐
                    │ [8] Zeroize   │  memset bootloader stack frame, locals
                    │   Secure      │  containing key material / header data
                    │   Locals      │
                    └──────┬────────┘
                           │
                           ▼
                    ┌───────────────┐
                    │ [9] BLXNS     │  Branch with Link to Non-Secure
                    │   → NS App    │  CPU transitions to NS state
                    └───────────────┘  Bootloader suspended; no return path
```

### 5.2 Image Validation Sequence (Step [4])

All six checks must pass. They execute in the order shown — later checks are not run if an earlier one fails.

```
[4a] Read header magic
     header.magic == 0x504B5346 ("PKSF")?
     NO  → FAIL (FAULT_IMAGE_INVALID)

[4b] Validate target ID
     header.target_id matches compile-time TARGET_ID (0x01=Node, 0x02=Gateway)?
     NO  → FAIL (FAULT_IMAGE_INVALID)

[4c] Anti-rollback counter check
     header.rollback_counter >= ota_ctrl.min_rollback_counter?
     header.rollback_counter >= secure_config.abs_min_rollback_counter?
     NO  → FAIL (FAULT_IMAGE_ROLLBACK)

[4d] Bootloader compatibility check
     header.min_btl_version_packed <= BTL_VERSION_PACKED (bootloader's own version)?
     NO  → FAIL (FAULT_BTL_INCOMPATIBLE)

[4e] Compute SHA-256 digest (via HASH peripheral)
     Zero: header.signature[64] (0x20), header.sha256_digest[32] (0x70), header._padding[112] (0x90)
     Input: header bytes [0x00..0x1F] ‖ image body (header.image_size bytes at slot_base + 0x100)
     Compare computed digest to header.sha256_digest — fast corruption check before PKA
     MISMATCH → FAIL (FAULT_IMAGE_INVALID)

[4f] Verify ECDSA P-256 signature (via PKA peripheral)
     public_key ← secure_config.ecdsa_public_key (key matching header.signing_key_id)
     Verify header.signature[64] against SHA-256 digest from [4e] using PKA
     FAIL → FAIL (FAULT_SIGNATURE_INVALID)

[4g] Verify CRC-32/MPEG-2
     Compute CRC-32 over image body; compare to header.crc32
     MISMATCH → FAIL (FAULT_IMAGE_CRC) [guards against bit-flip post-signing; SYS-S-004]

ALL PASS → image is authentic, integrity-verified, target-matched, and authorized
```

**Why this order?**
- [4a]–[4b]: Byte comparisons — reject garbage or cross-target images immediately.
- [4c]–[4d]: Integer comparisons — reject rollbacks and incompatible images before any crypto.
- [4e]: SHA-256 (~10 ms) — fast integrity confirmation before the slower PKA operation.
- [4f]: ECDSA P-256 (~12 ms) — authenticity proof; can reuse digest from [4e].
- [4g]: CRC-32 (~5 ms) — redundant integrity check required by SYS-S-004; catches bit-flips ECDSA hash would also catch but runs on a separate code path.

### 5.3 Failure Handling

| Candidate | Failure step | Action |
|-----------|-------------|--------|
| App B | Any of [4a]–[4f] | Write `fault_record_t`. Clear `ota_ctrl.swap_magic`. Re-run validation with App A as candidate. |
| App A | Any of [4a]–[4f] | Write `fault_record_t(FAULT_IMAGE_INVALID)`. Enter `while(1U) {}`. IWDG fires after 4 s → reset. On next boot, App A fails again → permanent boot loop until reflashed via SWD. |
| App A (after App B fallback) | Any of [4a]–[4f] | Same as App A above. |

**Why halt-then-IWDG rather than reset immediately?**
The `while(1U)` gives the fault write time to complete and prevents a rapid reboot storm that could wear the Fault Log flash. The IWDG is already armed at 4 s — it provides the reset automatically. No explicit reset call is needed from error paths.

**App A as permanent fallback:** App A is never erased by the bootloader. It is only erased by `ota.c` in the application layer at the start of the *next* OTA download cycle, after the running App B has proven stable (e.g., after N successful uptime cycles — defined in [[5.15-application-design]]).

## 6. Data Structures

> **Authoritative definitions are in [[5.09-module-headers-and-metadata]].** This section summarises the fields the bootloader directly accesses and the logic it applies to them.

### 6.1 `ps_image_header_t` — Application Image Header

First 256 bytes of each application slot. Full C struct and field map: [[5.09-module-headers-and-metadata]] §2.

**Fields read by `btl_validate.c`:**

| Offset | Field | Bootloader action |
|--------|-------|------------------|
| 0x00 | `magic` | Compare to `PS_IMAGE_MAGIC` (0x504B5346) |
| 0x04–0x07 | `major`, `minor`, `patch` | Informational only (rollback uses `rollback_counter`) |
| 0x08 | `rollback_counter` | Compare to `ota_ctrl.min_rollback_counter` and `secure_config.abs_min_rollback_counter` |
| 0x0C | `image_size` | Length of body fed to SHA-256 and CRC-32 |
| 0x10 | `crc32` | Expected CRC-32 of body; verified at step [4g] |
| 0x18 | `target_id` | Must match compile-time `TARGET_ID` |
| 0x1A | `signing_key_id` | Selects which public key to use from Secure Config |
| 0x1C | `min_btl_version_packed` | Must be ≤ `BTL_VERSION_PACKED` |
| 0x20 | `signature[64]` | ECDSA P-256 r‖s; verified via PKA at step [4f] |
| 0x70 | `sha256_digest[32]` | Pre-computed digest; compared to recomputed at step [4e] |

**Signing coverage:** `SHA-256(header[0x00–0x1F] ‖ image_body)` with `signature`, `sha256_digest`, and `_padding` zeroed before hashing.

### 6.2 `ota_control_t` — OTA Control Sector

Located at `0x081E2000`, page 113, Bank 2 (8 KB, NS). Full definition: [[5.09-module-headers-and-metadata]] §3.

**Key change from initial design:** Anti-rollback uses `min_rollback_counter` (a dedicated monotonic integer from `header.rollback_counter`), not `min_version_packed`. Semantic version comparisons are ambiguous for patch releases on older branches; a dedicated counter is not.

**Fields read and written by bootloader:**

| Field | Read | Written at commit |
|-------|------|------------------|
| `swap_magic` | Slot selection at step [3] | Cleared to `0x00000000` |
| `committed` | Checked to skip re-validation on clean boot | Set to `0x00000001` |
| `min_rollback_counter` | Anti-rollback check at step [4c] | Updated to `header.rollback_counter` if higher |
| `crc32_ctrl` | Struct integrity at step [3] | Recomputed after commit write |

### 6.3 `secure_config_t` — Secure Configuration Sector (Bootloader-Relevant Fields)

Located at `0x081E0000` (Secure, SECWM2). Full definition: [[5.09-module-headers-and-metadata]] §4.

**Fields read by bootloader:**

| Field | Offset | Purpose |
|-------|--------|---------|
| `magic` | 0x00 | Must be `0x53434647` "SCFG" |
| `crc32_cfg` | 0x04 | CRC-32 over struct bytes [0x08..end]; verified before any key use |
| `ecdsa_public_key[64]` | 0x08 | ECDSA P-256 public key Qx‖Qy for signature verification |
| `active_signing_key_id` | 0x4C | Must match `header.signing_key_id`; guards against key confusion |
| `abs_min_rollback_counter` | 0x50 | Absolute anti-rollback floor; cannot be lowered without RDP regression |

If `magic` or `crc32_cfg` fails: `fault_record_t(FAULT_SCFG_CORRUPT)` → `while(1U)`.

### 6.4 `fault_record_t` — Fault Log Entry

```c
#define FAULT_LOG_BASE      0x081FE000UL
#define FAULT_LOG_RING_SIZE 0x00001000UL   /* 4 KB = 128 records */
#define FAULT_RECORD_MAGIC  0xDEADBEEFUL
#define FAULT_RECORD_SIZE   32U

typedef struct
{
    uint32_t     magic;         /* 0x00: FAULT_RECORD_MAGIC                */
    fault_code_t code;          /* 0x04: fault_code_t from error_codes.h   */
    uint32_t     cfsr;          /* 0x08: CFSR register at fault point       */
    uint32_t     pc;            /* 0x0C: Faulting PC (from exception frame) */
    uint32_t     lr;            /* 0x10: Link register at fault             */
    uint32_t     timestamp_s;   /* 0x14: RTC seconds since epoch            */
    uint8_t      _pad[8U];      /* 0x18: Zeroed; aligns to 32 bytes         */
} fault_record_t;               /* Total: 32 bytes; 128 fit in 4 KB ring    */
```

**Ring buffer management:** The first 4 bytes of the Fault Log page (`0x081FE000`) are a `uint32_t write_index` (record count mod 128). A new record is written at `FAULT_LOG_BASE + 4 + (write_index % 128U) * 32U`, then `write_index` is incremented and rewritten. This allows 128 records before overwriting the oldest.

**Fault log is NS:** The fault log is in NS flash (page 127). The bootloader writes to it via direct flash programming (the bootloader runs in Secure world and can access NS flash). The NS application reads the fault log via its normal NS flash access. Post-mortem analysis is available without SWD.

## 7. Anti-Rollback Mechanism

### 7.1 Rollback Counter vs. Semantic Version

Anti-rollback is enforced via `header.rollback_counter` — a dedicated 32-bit monotonic integer — rather than the semantic version triple. Reason: semantic versioning is ambiguous for rollback detection. A team may release `v2.0.3` (a security patch on the 2.0 branch) after shipping `v2.1.0`, making `v2.0.3 < v2.1.0` by version comparison — yet `v2.0.3` is the *newer* security build for devices still on the 2.0 branch. The rollback counter has no such ambiguity: it only increases, regardless of branch.

### 7.2 Two-Layer Anti-Rollback

| Layer | Field | Floor set by | Can decrease? |
|-------|-------|-------------|---------------|
| **OTA layer** | `ota_ctrl.min_rollback_counter` | Bootloader at each commit | No (bootloader only writes `max(existing, new)`) |
| **Absolute layer** | `secure_config.abs_min_rollback_counter` | Manufacturing / reprovision | No (requires RDP regression + chip erase) |

Both checks run at step [4c]. An image must satisfy both floors simultaneously.

### 7.3 Enforcement Rules

1. `header.rollback_counter >= ota_ctrl.min_rollback_counter` — enforced at step [4c] before any crypto.
2. `header.rollback_counter >= secure_config.abs_min_rollback_counter` — enforced at step [4c].
3. Rejection happens before SHA-256 / ECDSA — a rollback image never reaches the PKA.
4. After committing App B, the bootloader writes `ota_ctrl.min_rollback_counter = header.rollback_counter` (never lower than the previous value).
5. The CI/CD pipeline enforces `ROLLBACK_COUNTER >= ota_ctrl.min_rollback_counter` as a pre-release gate, not just the bootloader.

### 7.4 Counter Management

The `rollback_counter` value is stored in `CMakeCache.txt` as `ROLLBACK_COUNTER` and injected into the header by `tools/inject_header.py`. Rules:
- Increment when releasing a security fix or any breaking change.
- Never decrement, even for a patch on an older branch.
- The CI/CD pipeline fails the build if `ROLLBACK_COUNTER` is absent from the cache.

## 8. Hardware Security Checks at Boot

The following checks are performed before any application code runs. All are in `BSP_Secure_Init()` or `btl_main.c`.

| # | Check | Pass condition | Fail action |
|---|-------|---------------|-------------|
| 1 | Flash latency appropriate for 160 MHz | `FLASH_ACR.LATENCY == 4` | Set it; no fault — this is config, not a tamper signal |
| 2 | TZEN option byte | `FLASH_OPTR.TZEN == 1` | `fault_log_write(FAULT_TZEN_DISABLED)` → halt |
| 3 | SECWM1 option bytes | `STRT=0, END=15` | `fault_log_write(FAULT_SECWM_TAMPER)` → `NVIC_SystemReset()` |
| 4 | SECWM2 option bytes | `STRT=112, END=112` | `fault_log_write(FAULT_SECWM_TAMPER)` → `NVIC_SystemReset()` |
| 5 | WRP1 option bytes | `WRP1A_STRT=0, WRP1A_END=15` | `fault_log_write(FAULT_WRP_TAMPER)` → `NVIC_SystemReset()` |
| 6 | SAU configured | Verify SAU_CTRL.ENABLE=1, ALLNS=0, all 6 regions match expected base/limit/type | `fault_log_write(FAULT_SAU_TAMPER)` → `NVIC_SystemReset()` |
| 7 | GTZC MPCBB3 (SRAM3) | All blocks Secure | `fault_log_write(FAULT_GTZC_TAMPER)` → `NVIC_SystemReset()` |
| 8 | GTZC peripheral security | AES, PKA, RNG are Secure | `fault_log_write(FAULT_GTZC_TAMPER)` → `NVIC_SystemReset()` |
| 9 | Secure Config struct integrity | `magic == 0x53434647` and CRC-32 valid | `fault_log_write(FAULT_SCFG_CORRUPT)` → halt |

Checks 3–8 use `NVIC_SystemReset()` (not halt) because they may indicate a transient power issue rather than an attack. After three consecutive resets with the same fault code in the fault log (detected by reading the last N records on boot), the bootloader switches to halt policy.

## 9. Hardware Fault Handlers

The bootloader installs its own vector table at `0x08000000`. All fault handlers follow the same pattern:

```
HardFault / MemManage / BusFault / UsageFault
  │
  ├── 1. Save exception frame (hardware-stacked): PC, LR, xPSR, R0–R3, R12
  ├── 2. Read CFSR (0xE000ED28)
  ├── 3. Call btl_fault_record_write(fault_code, CFSR, stacked_PC, stacked_LR)
  └── 4. while(1U) {}   ← IWDG fires after 4 s → reset
```

```c
/* Assembly stub for HardFault — extracts stacked PC and LR from MSP */
void HardFault_Handler(void)
{
    __asm volatile (
        "TST    LR, #4          \n"   /* Check EXC_RETURN bit 2 */
        "ITE    EQ              \n"
        "MRSEQ  R0, MSP         \n"   /* Secure stack was active */
        "MRSNE  R0, PSP         \n"
        "B      btl_hard_fault_c\n"   /* R0 = pointer to stacked frame */
    );
}

void btl_hard_fault_c(const uint32_t *p_frame)
{
    uint32_t pc   = p_frame[6U];   /* Stacked PC  */
    uint32_t lr   = p_frame[5U];   /* Stacked LR  */
    uint32_t cfsr = SCB->CFSR;
    btl_fault_record_write(FAULT_HARDFAULT, cfsr, pc, lr);
    while (1U) {}
}
```

## 10. ECDSA Verification Implementation Notes

### 10.1 Key Source

The 64-byte ECDSA P-256 uncompressed public key (Qx‖Qy) is read from `secure_config.ecdsa_public_key` at `0x081E0000 + 0x08`. The bootloader reads it directly (Secure world, SECWM2 makes this region Secure-only). No veneer is needed — the veneer is for NS→Secure calls, not Secure-to-Secure access.

### 10.2 PKA Operation Sequence

```
1. Enable PKA clock (RCC_AHB2ENR.PKAEN)
2. Reset PKA (PKA_CR.RESET)
3. Set PKA mode: ECDSA verify (PKA_CR.MODE = 0x26)
4. Write PKA input RAM:
     - Prime p, order n, cofactor a, generator G (curve P-256 constants)
     - Public key Qx, Qy (64 bytes from Secure Config)
     - Message hash (SHA-256 digest, 32 bytes)
     - Signature r (bytes [0..31] of header.signature)
     - Signature s (bytes [32..63] of header.signature)
5. Start PKA (PKA_CR.START = 1)
6. Poll PKA_SR.PROCENDF until set (or timeout 500 ms)
7. Read PKA_RAM result: PKA_ECDSA_VERIF_OUT_RESULT
     0x00000000 = verification OK
     0xFFFFFFFF = verification failed
8. Disable PKA clock
```

Timeout on PKA poll: if 500 ms elapses without `PROCENDF`, write `fault_record_t(FAULT_PKA_TIMEOUT)` and treat as verification failure. A 500 ms budget is generous — ECDSA P-256 verify on STM32U585 at 160 MHz takes approximately 8–12 ms.

### 10.3 SHA-256 Digest Computation

```
1. Enable HASH clock
2. Configure HASH: SHA-256 mode, input DMA or polling
3. Feed input: header[0x00..0x1F] with signature bytes zeroed (32 bytes)
4. Feed input: image body (header.image_size bytes starting at slot_base + 0x100)
5. Wait for DCIS flag (digest computation complete)
6. Read digest[0..7] (8 × 32-bit words = 32 bytes)
```

The image body is fed in 32-byte aligned chunks via polling (DMA not available in bootloader without RTOS). Maximum image size 896 KB at 160 MHz takes approximately 5–10 ms.

## 11. Boot Timing Budget

| Step | Typical Duration | Notes |
|------|----------------|-------|
| BSP_Secure_Init (clocks/SAU/GTZC/MPU) | ~3 ms | PLL lock ~1 ms; register writes ~2 ms |
| SECWM / WRP option byte read | < 1 ms | Register reads |
| OTA Control sector read + CRC | < 1 ms | 16 bytes |
| Header read (256 bytes) | < 1 ms | Single flash read |
| Magic / target / version checks | < 1 ms | Integer comparisons |
| SHA-256 over 896 KB max image | ~10 ms | HASH peripheral, polling |
| ECDSA P-256 verify | ~12 ms | PKA peripheral |
| CRC-32 over 896 KB max image | ~5 ms | Software (or CRC peripheral) |
| NS MPU config + VTOR_NS | < 1 ms | Register writes |
| **Total (worst case)** | **~33 ms** | Well within any reasonable power-on budget |

The IWDG 4 s timeout is not stressed by this budget. `BSP_IWDG_Pet()` is called once at the end of `BSP_Secure_Init()` to reset the 4 s window before validation begins.

## 12. OTA Swap Commit Details

### 12.1 Sequence After Successful App B Validation

```
1. Flash erase OTA Control page (0x081E2000, 8 KB)
2. Prepare new ota_control_t:
     swap_magic             = 0x00000000    (clear pending flag)
     committed              = 0x00000001
     min_rollback_counter   = max(ota_ctrl.min_rollback_counter, header.rollback_counter)
     crc32_ctrl             = CRC32(new_struct[0..0x0B])
```

### 12.2 App A Rollback Window

App A remains in flash as a fallback after an OTA commit. It is erased only when `ota.c` in the running application explicitly issues a flash erase of the App A slot at the start of the *next* OTA download cycle. Until then, if the new App B image crashes on every boot (e.g., stuck in HardFault), the IWDG resets the chip, the bootloader runs, sees `swap_magic == 0` and `committed == 1`, and boots App B again — the bootloader itself cannot undo a committed swap. App-level watchdog-based rollback (counting resets in SRAM4 before the app stack is overwritten) is described in [[5.15-application-design]].

## 13. Security Invariants

| # | Invariant | Enforcement Mechanism | SyRS Req |
|---|-----------|----------------------|----------|
| 1 | Bootloader code cannot be modified in the field | WRP1 option bytes lock pages 0–15 | SYS-S-001 |
| 2 | Bootloader binary is hidden from debugger and NS bus | HDP1 option bytes hide pages 0–15 | SYS-S-008 |
| 3 | ECDSA public key is protected from NS access and modification | SECWM2 + GTZC; Secure Config page write-locked (WRP2) in production | SYS-S-009 |
| 4 | NS application cannot invoke PKA, AES, or RNG directly | GTZC peripheral TZ registers mark accelerators as Secure | SYS-S-008 |
| 5 | NS application cannot execute code from App B slot | NS MPU region 1: XN=1 | SYS-S-008 |
| 6 | NS application cannot write to App A slot (self-modification) | NS MPU region 0: AP=RX only, write fault | SYS-S-008 |
| 7 | Firmware downgrade is impossible after commit | `min_rollback_counter` monotonically increases in OTA Control; absolute floor in `secure_config.abs_min_rollback_counter` | SYS-S-004 |
| 8 | ECDSA private key never reaches the device | Only public key in Secure Config; private key in GitHub Actions Secret / HSM | SYS-S-002 |
| 9 | Unsigned or tampered images are rejected before execution | ECDSA P-256 signature verified via PKA before `BLXNS` | SYS-S-001, SYS-S-003 |
| 10 | Images for wrong target cannot boot | `target_id` field checked at step [4b] | SYS-S-004 |
| 11 | All security configuration is verified on every boot | Checks §8 run before every `BLXNS` — not only at first boot | SYS-S-008 |
| 12 | Fault evidence survives reset | `fault_record_t` written to NS flash before any reset or halt | — |
