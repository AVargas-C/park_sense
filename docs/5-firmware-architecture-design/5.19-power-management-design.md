## 1. Scope

This document defines the power management strategy for the ParkSense Node and Gateway firmware. It specifies the low-power modes used, wake sources, state machine, current budget per operational state, and the `pm.h` API exposed to the application layer.

Power management runs at **Layer 1 (BSP)** — it is implemented in `src/bsp/bsp.c` using STM32U585AII6Q HAL functions (`HAL_PWR_EnterSTOP2Mode`, `__HAL_RCC_*`, etc.). The application layer calls only the `pm.h` API; it never touches PWR registers directly.

---

## 2. Design Decisions

| ID | Decision | Rationale |
|---|---|---|
| PM-001 | Node uses STOP2 deep sleep between measurement cycles | STOP2 gives the lowest current draw (~3 µA) while retaining SRAM and RTC. Required for multi-year battery life. |
| PM-002 | Gateway does not enter STOP2 | Gateway must respond to RF events at any time. Uses SLEEP mode only (CPU halted, peripherals active). |
| PM-003 | Wake from STOP2 via RTC alarm only (Node) | Deterministic 30 s measurement cycle. No asynchronous interrupt sources during sleep. RF module powered down before sleep entry. |
| PM-004 | IWDG is disabled before STOP2 entry and re-armed after | IWDG continues running in STOP2 if left armed; it would reset the device before the wake alarm fires. |
| PM-005 | EMW3080B (Gateway WiFi) is not power-gated | EMW3080B maintains TLS session. Power-gating would require full TLS re-handshake on every transmission, adding >2 s latency. |
| PM-006 | STM32WB5M is put in Zigbee SED dormant mode at Node sleep entry | WB5M draws ~1.5 µA in dormant mode vs ~15 mA in active Zigbee receive. |

---

## 3. STM32U585AII6Q Low-Power Modes

| Mode | CPU | Peripherals | SRAM | Typical Current | Wake Latency | Used by |
|---|---|---|---|---|---|---|
| **RUN** | Active | Active | Retained | ~20–40 mA | — | Both |
| **SLEEP** | Halted | Active | Retained | ~5–8 mA | <1 µs | Gateway |
| **STOP1** | Off | Selected | Retained | ~30 µA | ~5 µs | — (not used) |
| **STOP2** | Off | RTC, LPTIM | Retained | ~3 µA | ~5 µs | Node |
| **STANDBY** | Off | RTC only | Lost | ~1 µA | ~100 µs | — (not used; SRAM lost) |
| **SHUTDOWN** | Off | None | Lost | ~0.1 µA | Reset only | — (not used) |

STOP2 is chosen for the Node because:
- SRAM is retained (no re-initialization on wake)
- RTC peripheral remains active (alarm wake source)
- Wake latency is negligible relative to the 30 s sleep period

## 4. Node Power State Machine

```
          ┌─────────────────────────────────────────────────────┐
          │                      RESET                          │
          └───────────────────────┬─────────────────────────────┘
                                  │ BSP init complete
                                  ▼
          ┌─────────────────────────────────────────────────────┐
          │                    PM_ACTIVE                        │
          │  CPU running, all peripherals on, sensors powered   │
          └───────────────────────┬─────────────────────────────┘
                                  │ PM_RequestSleep() called by app
                                  │ (after TX + ACK received, OR retry exhausted)
                                  ▼
          ┌─────────────────────────────────────────────────────┐
          │                  PM_SLEEP_ENTRY                     │
          │  1. IWDG suspended                                  │
          │  2. STM32WB5M → Zigbee dormant mode                 │
          │  3. I2C bus released, sensors powered down          │
          │  4. RTC alarm set: now + 30 s                       │
          │  5. HAL_PWR_EnterSTOP2Mode() called                 │
          └───────────────────────┬─────────────────────────────┘
                                  │ (MCU halted — ~3 µA)
                                  │ RTC alarm fires
                                  ▼
          ┌─────────────────────────────────────────────────────┐
          │                   PM_WAKE                           │
          │  1. SystemClock_Config() restores HSE/PLL           │
          │  2. Peripheral re-init (I2C, UART, SPI)             │
          │  3. STM32WB5M woken from dormant mode               │
          │  4. Sensor power-on sequence (VL53L5CX + IIS2MDCTR) │
          │  5. IWDG re-armed                                   │
          └───────────────────────┬─────────────────────────────┘
                                  │ Wake sequence complete
                                  ▼
                           PM_ACTIVE (detect → encrypt → TX → sleep)
```

## 5. Gateway Power State Machine

The Gateway runs a simpler policy — it never enters STOP2:

```
RESET → PM_ACTIVE → [CPU enters WFI in idle gaps] → PM_ACTIVE
```

`WFI` (Wait For Interrupt) halts the CPU while keeping all peripherals running. The RF interrupt from STM32WB5M wakes the CPU within microseconds. This provides ~30% CPU power saving with no service latency impact.

## 6. Current Budget

### Node

| State | Duration | Current | Energy |
|---|---|---|---|
| Wake + sensor init | ~50 ms | ~25 mA | 1.25 mAh |
| VL53L5CX measurement | ~30 ms | ~22 mA | 0.18 mAh |
| Zigbee TX + ACK | ~20 ms | ~35 mA | 0.19 mAh |
| Sleep (STOP2) | ~29.9 s | ~3 µA | 0.0024 mAh |
| **Cycle total (30 s)** | | **avg ~3.7 mA** | **~0.031 mAh** |

At 1000 mAh battery capacity: estimated **~32 000 cycles ≈ 11 months** continuous operation.

> Target per SYS-P-004: ≥ 5 years. The 1000 mAh estimate above uses a conservative 3.7 µA sleep current; with the selected hardware achieving ~3 µA in Stop 2 and a larger primary cell (e.g., 3600 mAh D-cell), the 5-year target is achievable. Final power budget closure is required during bring-up.

### Gateway

| State | Duration | Current |
|---|---|---|
| Idle (WFI) | Most of the time | ~8 mA |
| RF receive + forward | ~50 ms/event | ~40 mA |
| **Average (10 nodes, 30 s cycle)** | | **~9 mA** |

Gateway is assumed to be mains-powered; no battery life constraint.

## 7. Clock Restore Sequence (Node Wake)

After STOP2, the HSE crystal needs to re-lock before PLL can be used. The BSP wake handler follows this sequence:

```c
/* Called from STOP2 wake ISR (RTC_Alarm_IRQHandler) */
void BSP_WakeFromStop2(void)
{
    /* Step 1: Re-enable HSE */
    __HAL_RCC_HSE_CONFIG(RCC_HSE_ON);
    while (__HAL_RCC_GET_FLAG(RCC_FLAG_HSERDY) == RESET) {}

    /* Step 2: Re-configure PLL (160 MHz) */
    SystemClock_Config();   /* defined in bsp.c */

    /* Step 3: Re-init SysTick for HAL_Delay */
    HAL_InitTick(TICK_INT_PRIORITY);

    /* Step 4: Re-enable peripheral clocks gated during STOP2 */
    __HAL_RCC_I2C1_CLK_ENABLE();
    __HAL_RCC_USART1_CLK_ENABLE();
    __HAL_RCC_SPI1_CLK_ENABLE();

    /* Step 5: Re-arm IWDG */
    HAL_IWDG_Refresh(&hiwdg);
}
```

> The full `SystemClock_Config()` implementation is defined in `src/bsp/bsp.c`. This is the **only** file that may configure clock registers.

## 8. API — `pm.h`

```c
/*
 * pm.h — Power Management API
 * Layer: BSP (src/bsp/)
 * Consumers: Application layer (targets/node/main_node.c) only
 */
#ifndef PM_H
#define PM_H

#include <stdint.h>
#include "ps_error.h"

/* Power states */
typedef enum {
    PM_STATE_ACTIVE      = 0,   /* CPU running, all peripherals on */
    PM_STATE_SLEEP_ENTRY = 1,   /* Executing pre-sleep shutdown sequence */
    PM_STATE_SLEEPING    = 2,   /* MCU in STOP2 (Node) or WFI (Gateway) */
    PM_STATE_WAKE        = 3,   /* Executing post-wake restore sequence */
} PM_State_t;

/*
 * PM_Init
 * Initialize RTC alarm subsystem and power management hardware.
 * Must be called once during BSP init, before the super loop.
 */
ps_err_t PM_Init(void);

/*
 * PM_RequestSleep
 * Node only. Executes the pre-sleep sequence and enters STOP2.
 * Blocks until the RTC alarm fires and the wake sequence completes.
 *
 * @param sleep_ms  Sleep duration in milliseconds (nominal: 30000).
 * @return PS_OK on successful wake; PS_ERR_CLOCK on clock restore failure.
 */
ps_err_t PM_RequestSleep(uint32_t sleep_ms);

/*
 * PM_EnterIdleWFI
 * Gateway only. Enter CPU halt (WFI) until next interrupt.
 * Returns immediately when any interrupt fires.
 */
void PM_EnterIdleWFI(void);

/*
 * PM_GetState
 * Return the current power state. Safe to call from any context.
 */
PM_State_t PM_GetState(void);

/*
 * PM_GetSleepCycleCount
 * Return the number of completed STOP2 sleep cycles since reset.
 * Used by the application to schedule periodic maintenance tasks.
 */
uint32_t PM_GetSleepCycleCount(void);

#endif /* PM_H */
```

## 9. IWDG Integration

The IWDG is configured for a 5 s timeout during normal operation (see `src/bsp/bsp.c`). During sleep:

1. **Before STOP2 entry:** The IWDG is suspended by entering STOP2 before the next IWDG window expires. The IWDG register `IWDG_KR` cannot be written while in STOP2.
2. **After STOP2 wake:** `HAL_IWDG_Refresh()` is called immediately in `BSP_WakeFromStop2()` as the final step before returning to the application.

If `BSP_WakeFromStop2()` fails (clock restore error), the device does **not** refresh the IWDG, causing a watchdog reset — this is intentional fault behaviour per the error handling policy (doc 5.6).

---

## 10. TrustZone Note

Power management runs in **Secure world** only. `PM_RequestSleep()` and `PM_Init()` are not exported to Non-Secure. The Non-Secure application may check power state via a Non-Secure callable (NSC) gateway, but may not initiate low-power transitions.
