## Phase 1 — ARM Architecture Selection

The first decision is which ARM architecture family meets the combined requirements for security isolation, crypto, DSP/FPU, and ultra-low power.

### Candidates

| Feature                                             | ARMv6-M (Cortex-M0/M0+) | ARMv7-M / ARMv7E-M (Cortex-M3/M4/M7) | ARMv8-M (Cortex-M23/M33/M35P/M55/M85) |
| --------------------------------------------------- | ------------------------ | -------------------------------------- | -------------------------------------- |
| **TrustZone hardware isolation**                    | No                       | No                                     | **Yes** — native to architecture       |
| **MPU**                                             | Optional (limited)       | Yes (8 regions)                        | **Yes** — enhanced (up to 16 regions)  |
| **DSP extensions**                                  | No                       | M4/M7 only (ARMv7E-M)                 | **Yes** — M33, M35P, M55, M85          |
| **Single-precision FPU**                             | No                       | M4/M7 optional                         | **Yes** — M33+ optional                |
| **Hardware crypto support (implementation via SoC)** | Rare                     | Available in some SoCs                 | **Standard** in ultra-low-power SoCs   |
| **Security Attribution Unit (SAU)**                  | No                       | No                                     | **Yes** — required for TrustZone       |
| **Stack limit checking**                             | No                       | No                                     | **Yes** — hardware stack overflow protection |
| **Ultra-low-power SoC availability**                 | Good (M0+)               | Good (M4)                              | **Excellent** — newest ULP families    |
| **Bit-width**                                        | 32-bit                   | 32-bit                                 | 32-bit                                 |

### Decision: ARMv8-M

**ARMv6-M eliminated:** No TrustZone, no MPU (or limited), no DSP, no FPU. Cannot satisfy SYS-S-008.

**ARMv7-M / ARMv7E-M eliminated:** No TrustZone. While Cortex-M4/M7 offer DSP and FPU, they lack the hardware security architecture (SAU, TrustZone, stack limit checking) required by SYS-S-008, SYS-S-009, and the overall cybersecurity architecture. Software-emulated isolation would violate the hardware-enforced isolation constraint.

**ARMv8-M selected:** The only ARM architecture with native TrustZone, SAU, enhanced MPU, and hardware stack limit checking — while also supporting DSP and FPU extensions. This architecture is the foundation for the newest ultra-low-power MCU families from all major vendors.

## Phase 2 — ARMv8-M Core Selection

Within ARMv8-M, five cores are available. The table below compares features **beyond** what is already common to all ARMv8-M cores (TrustZone, SAU, enhanced MPU, stack limit checking).

| Feature                                    | Cortex-M23         | Cortex-M33          | Cortex-M35P         | Cortex-M55             | Cortex-M85             |
| ------------------------------------------ | ------------------ | ------------------- | ------------------- | ---------------------- | ---------------------- |
| **Pipeline**                               | 2-stage            | 3-stage             | 3-stage             | 4–5-stage              | 6–7-stage              |
| **Architecture profile**                   | Baseline           | **Mainline**        | Mainline            | Mainline               | Mainline               |
| **DSP extensions**                         | No                 | **Yes** (optional)  | Yes                 | Yes + MVE (Helium)     | Yes + MVE (Helium)     |
| **FPU**                                    | No                 | **SP optional**     | SP optional         | SP + DP + MVE          | SP + DP + MVE          |
| **Co-processor interface**                 | No                 | **Yes** (up to 8)   | Yes                 | Yes                    | Yes                    |
| **Physical memory protection (anti-tamper)**| No                 | No                  | **Yes** — cache with physical address tagging | No          | No                     |
| **Helium (M-profile Vector Extension)**    | No                 | No                  | No                  | **Yes**                | **Yes**                |
| **Typical clock speed**                    | ≤ 48 MHz           | **≤ 160 MHz**       | ≤ 200 MHz           | ≤ 480 MHz              | ≤ 600 MHz              |
| **Target applications**                    | Cost-sensitive IoT  | **IoT, industrial, security** | Tamper-resistant | AI/ML at the edge   | High-perf. real-time   |
| **Ultra-low-power SoCs available**         | Limited             | **Excellent**       | Very limited        | Limited                | Very limited           |
| **Power consumption class**                | Lowest (minimal features) | **Low**        | Moderate            | Higher                 | Highest                |
| **SoC ecosystem maturity**                 | Moderate            | **Mature**          | Niche               | Growing                | Emerging               |

### Decision: Cortex-M33

**Cortex-M23 eliminated:** Baseline profile — no DSP, no FPU, no co-processor interface. Sensor fusion (SYS-F-003) requires DSP/FPU for efficient distance and field-strength calculations. Cannot meet processing constraints.

**Cortex-M35P eliminated:** Designed for physical tamper resistance (anti-side-channel cache). Valuable for payment terminals and secure elements, but adds silicon cost and complexity not justified for a parking sensor. Near-zero ultra-low-power SoC availability.

**Cortex-M55 eliminated:** Helium/MVE extensions are designed for AI/ML and DSP-heavy signal processing. Massively overpowered for a duty-cycled parking sensor. Higher power consumption defeats battery life target (SYS-P-004). Limited ULP SoC availability.

**Cortex-M85 eliminated:** Highest-performance ARMv8-M core, targeting real-time control at 600 MHz. Extreme overkill for a 30-second wake cycle. No ultra-low-power SoCs available.

**Cortex-M33 selected:** Mainline profile with optional DSP and FPU — exactly the feature set required for sensor fusion. The most mature ultra-low-power SoC ecosystem of any ARMv8-M core. Balances security (TrustZone + SAU), processing (DSP + FPU), and power efficiency. Every major MCU vendor offers Cortex-M33 parts targeting battery-powered IoT.

### Key Observations

**STM32U585 vs STM32U575:**
- Identical silicon, identical low-power modes. The U585 adds hardware AES, SHA, PKA, and OTFDEC. The U575 lacks these crypto accelerators entirely. Since SyRS requires HW AES (SYS-S-005), HW PKA (SYS-S-002 — asymmetric signature verification), and SHA (SYS-S-004 — firmware integrity), the **U575 cannot satisfy security requirements** without external crypto hardware.

**STM32L552/L562:**
- Previous-generation ultra-low-power Cortex-M33. Lower flash (max 512 KB — at the minimum threshold), lower clock (110 MHz), and significantly higher active power (62 vs 19 µA/MHz). The L562 variant has crypto accelerators but the overall power/performance ratio is inferior to the U5 series.

**nRF54L15:**
- Integrated radio is a major advantage — eliminates the need for a separate RF module. However: the node architecture requires separate sensor (I2C) and RF modules to maintain Layer 3 portability (SYS-NF-001). Using the integrated radio tightly couples the MCU and RF selection, violating the portability principle. Also, the nRF Connect SDK / Zephyr RTOS ecosystem is oriented toward an RTOS operating model rather than bare-metal super loop, adding unnecessary complexity for this application.

**EFR32MG24:**
- Similar trade-off to nRF54L15: integrated radio creates hardware/firmware coupling. EM2 stop mode current is competitive, but active power is higher than STM32U5. Gecko SDK ecosystem is less mature for bare-metal Cortex-M33 development.

## Selection Rationale

### Selected: STM32U585AII6Q

**Primary decision driver: Gateway crypto throughput + codebase unification.**

The STM32U575 (without hardware crypto) was considered as a cost-saving option for the IoT sensor node. At 160 MHz, the software AES encryption time for a single small occupancy packet is negligible — the performance impact on the node's super loop cycle is immaterial. For the node alone, the U575 would be sufficient.

However, the **gateway** is the bottleneck. When receiving and decrypting packets from 100+ nodes in rapid succession, hardware-accelerated crypto becomes critical to maintain SYS-P-006 (forward packet within 500 ms) and avoid queuing delays under load. The U585's dedicated crypto accelerators handle this without consuming CPU cycles, leaving the Cortex-M33 core free for protocol processing and WiFi forwarding.

The U585 is the best fit because it includes the full suite of hardware security accelerators: **AES** (payload encryption/decryption), **PKA** (asymmetric key operations for secure key exchanges), **SHA** (hashing for firmware integrity checks), and **OTFDEC** (on-the-fly decryption for secure external memory). These accelerators are essential for digital signature validation during secure boot (SYS-S-001), secure key exchange during node commissioning (SYS-S-012), and runtime payload encryption (SYS-S-005).

Additionally, both U5 variants feature **LPBAM (Low-Power Background Autonomous Mode)** — a unique STM32U5 capability that allows DMA-driven peripheral transfers while the core remains in Stop mode. This enables autonomous I2C sensor reads or SPI transactions without waking the CPU, further reducing active time and extending battery life.

> **Note on U575:** While the STM32U575 is more budget-friendly (~$3–4 less per unit), the cost difference is not meaningful for a hobby/portfolio project. The U585's hardware accelerators provide tangible firmware simplification and security guarantees that outweigh the marginal savings.

**Decision: use the STM32U585 for both node and gateway.**

| Factor                                   | Rationale                                                                                                         |
| ---------------------------------------- | ----------------------------------------------------------------------------------------------------------------- |
| Gateway requires HW crypto               | 100+ concurrent nodes demand hardware-accelerated decryption to sustain throughput without CPU bottleneck          |
| Node could use SW crypto (U575)          | Single-packet AES at 160 MHz has negligible impact on the 1 s cycle budget — U575 would technically work          |
| Codebase unification wins                | One MCU target eliminates a second HAL/BSP configuration, a second CubeMX project, and a second CI build matrix   |
| Reduced supply chain risk                | Single part number to source, stock, and qualify                                                                   |
| Dev kit advantage                        | The B-U585I-IOT02A Discovery Kit includes onboard ToF sensor, magnetometer, WiFi, and BLE — ideal for prototyping |
| LPBAM advantage                          | Low-Power Background Autonomous Mode enables DMA peripheral transfers in Stop mode — unique to STM32U5 family    |
| HW accelerators (AES, PKA, SHA, OTFDEC)  | Digital signature validation, secure key exchange, payload crypto, and on-the-fly decryption — all offloaded to HW |
| U575 out of scope                        | U575 is more budget-friendly but the savings are not meaningful for a hobby/portfolio project                      |
