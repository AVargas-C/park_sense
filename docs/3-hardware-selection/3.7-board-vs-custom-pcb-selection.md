# 3.7 Board vs Custom PCB Selection

> **Project:** ParkSense — Full-Stack IoT Parking Occupancy System
> **Date:** 2026-02-21
> **Author:** Arturo Vargas Cuevas
> **↑ Parent:** [[3-Hardware selection and trade-off analysis]]

---

## Context

ParkSense is a portfolio and learning project, not a production deployment. This section documents the platform decision: whether to use an off-the-shelf development board, or design a custom PCB from the ground up.

This decision is upstream of all other hardware selections because the chosen platform determines which sensors, modules, and MCU are available for prototyping — and which require external hardware bring-up.

---

## Phase 1 — Development Board vs Custom PCB

### Comparison

| Factor                            | Development Board                                      | Custom PCB                                              |
| --------------------------------- | ------------------------------------------------------ | ------------------------------------------------------- |
| **Time to first firmware run**    | Hours — board is validated by manufacturer             | Weeks to months — schematic → layout → fab → assembly → bring-up |
| **Hardware debugging burden**     | Low — known-good hardware; bugs are in firmware        | High — hardware and firmware bugs are concurrent        |
| **Sensor integration**            | Pre-wired, pre-validated on board                      | Manual schematic, routing, decoupling, I2C pull-ups     |
| **Power supply design**           | Onboard regulators validated                           | Must design LDO / SMPS, battery management, filtering   |
| **RF module integration**         | Module on board or standard footprint                  | Antenna design, RF trace impedance matching, certification |
| **Schematics / BOM**              | Published by manufacturer                              | Must be authored and reviewed                           |
| **PCB manufacturing cost**        | N/A (included in board price)                          | $50–500 for prototype run (3–5 boards, JLCPCB / PCBWay) |
| **Component sourcing**            | N/A                                                    | Each component must be sourced, priced, validated        |
| **PCBA assembly (SMD)**           | N/A                                                    | Manual (skill-intensive) or SMT service ($100–300)      |
| **Regulatory certification**      | FCC / CE done by manufacturer                          | Full certification required (costly, multi-month)        |
| **Portfolio signal**              | Demonstrates firmware and system architecture depth    | Demonstrates HW design depth; adds PCB design as skill  |
| **Recommended for**               | Firmware development, portfolio demonstration          | Production hardware, physical form factor constraints   |

### Decision: Development Board

For ParkSense, the primary deliverable is **firmware architecture** — the layered firmware model, security stack, CI/CD pipeline, and cloud integration. A custom PCB would shift significant time into hardware bring-up and PCB debugging, displacing time available for firmware depth.

A development board with validated hardware allows the full firmware stack (bootloader → drivers → PDM/CPM → application → server → UI) to be completed and clearly demonstrated. Hardware design can be documented as a future phase or separate portfolio milestone.

## Selection Rationale

### Selected: B-U585I-IOT02A Discovery Kit

**Primary decision driver: the only board that integrates all required hardware for both sensor node and gateway prototyping on the confirmed MCU (STM32U585), with ST official driver support for every peripheral.**

| Factor                                   | Rationale                                                                                                       |
| ---------------------------------------- | --------------------------------------------------------------------------------------------------------------- |
| All-in-one integration                   | ToF (VL53L5CX), magnetometer (IIS2MDCTR), WiFi (EMW3080), BLE/Thread/Zigbee (STM32WB5M) — all pre-wired by ST  |
| STM32U585AII6Q                           | Exact MCU selected in 3.1 — TrustZone, HW AES/SHA/PKA, LPBAM, 160 MHz, 2 MB flash                             |
| Zero hardware bring-up cost              | All sensors are validated, documented, and exercised by ST's own firmware examples                               |
| ST official drivers for all peripherals  | VL53L5CX ULD, IIS2MDC driver, X-CUBE-MXCHIP WiFi, STM32WB BLE stack — no third-party porting                   |
| Dual-role prototyping (node + gateway)   | One board can prototype both sensor node firmware and gateway firmware by swapping compiled target               |
| Portfolio signal                         | Demonstrates ability to navigate a multi-peripheral embedded platform and produce structured firmware on real hardware |

> **Note on STEVAL-MKBOXPROA as a secondary node platform:** The STEVAL-MKBOXPROA is a strong alternative for sensor node prototyping. It uses the same MCU die (STM32U585AII6), includes an onboard LiPo battery enabling untethered deployment simulation, BlueNRG-355AC for BLE 5.4, and LIS2MDL magnetometer. The absence of a ToF sensor is architecturally meaningful: firmware compiled in magnetometer-only PDM mode demonstrates SYS-NF-001 (Layer 3 portability) — Layers 4–5 and CPM require no changes. This makes the STEVAL-MKBOXPROA a valuable secondary board for teaching PDM sensor portability.

### Honorable Mentions

#### All-in-One Discovery Kits

These platforms would have been viable alternatives if hardware-agnostic requirements had led to a different MCU or RF stack selection:

| Board               | Sensor / MCU                          | Why not selected                                                                             |
| ------------------- | ------------------------------------- | -------------------------------------------------------------------------------------------- |
| **nRF54L15-DK**     | nRF54L15, BLE 5.4 / Thread / Zigbee  | No WiFi onboard, no onboard ToF/mag, nRF Connect SDK is RTOS-first — conflicts with bare-metal super loop |
| **STEVAL-MKBOXPROA** | STM32U585AII6 (same die as B-U585I-IOT02A), LIS2MDL magnetometer, BlueNRG-355AC (BLE 5.4 ULP), onboard LiPo battery | **Strong candidate for node prototyping.** Same MCU as B-U585I-IOT02A. BLE 5.4 and magnetometer onboard. Onboard LiPo enables untethered operation. No ToF sensor — this demonstrates Layer 3 portability (SYS-NF-001): PDM runs in magnetometer-only mode with no changes to Layers 4–5 or CPM. WiFi must be added externally (SPI) for gateway role. |
| **STEVAL-BCN002V1** | STM32F401VE, LIS3MDL, LSM6DSM        | Cortex-M4 (no TrustZone), no HW crypto accelerators, BLE 4.2 only, STM32F4 HAL — does not satisfy security requirements |

#### Development Kit + External GPIO Modules

This approach (Nucleo board + breakout modules) is the most flexible in terms of pin access and is preferred for hardware design learning exercises. It would be recommended as a secondary platform for students who want to practice wiring and bring-up:

| Platform                           | Notes                                                                                                       |
| ---------------------------------- | ----------------------------------------------------------------------------------------------------------- |
| **NUCLEO-U575ZI-Q + sensor shields**| U575 lacks HW crypto — firmware AES required on node; otherwise a clean bare-metal STM32U5 starting point  |
| **NUCLEO-U585AI + sensor shields** | Full STM32U585 feature set (HW crypto, LPBAM) on a Nucleo form factor — all sensors must be added externally |
| **NUCLEO-L552ZE-Q (L562 variant) + sensors** | HW crypto on L562, but older generation (110 MHz, 62 µA/MHz active, no LPBAM) — viable for firmware architecture study |
| **nRF54L15-DK + sensor shields**   | Valid if choosing Nordic ecosystem; requires nRF Connect SDK / Zephyr, incompatible with bare-metal super loop model |

## Risk Assessment

| Risk Factor               | Assessment                                                                                                         |
| ------------------------- | ------------------------------------------------------------------------------------------------------------------ |
| Supply chain availability | **Low risk.** B-U585I-IOT02A is ST's active IoT demonstration platform. Available from major distributors.         |
| Hardware bring-up risk    | **None.** Board is manufactured and validated by ST. All sensors are pre-connected.                                 |
| Driver / SDK availability | **Low risk.** All peripheral drivers are available in STM32CubeU5 package maintained by ST.                        |
| Long-term support         | **Low risk.** STM32U5 is ST's current-generation ultra-low-power MCU family with expected 10+ year lifecycle.       |
| Scalability to production | **N/A.** Development boards are not production hardware. Custom PCB design would be the next phase for deployment.  |
