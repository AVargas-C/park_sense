## Phase 1 — Wireless Protocol Stack Comparison

Before selecting an RF module, the protocol stack must be chosen. The module selection depends on which protocol satisfies the combined requirements for security, reliability, and performance.

Candidate protocols are compared across three evaluation dimensions:

1. **Security** — encryption, integrity, replay protection, authentication, commissioning
2. **Reliability** — CRC, ACK/NACK, retransmission, PDR achievability
3. **Performance** — scalability, data rate, range, power, topology fit

**Protocols eliminated before detailed comparison:**

These protocols were rejected based on fundamental architectural mismatches with system requirements. A full three-way evaluation would have been wasted effort.

- **Thread:** Thread is mesh-first by design. Its IEEE 802.15.4 / 6LoWPAN / CoAP stack is RTOS-oriented — the OpenThread reference implementation requires a task scheduler (Zephyr, FreeRTOS, or equivalent) to drive MLE state machines, mesh routing tables, and CoAP/UDP handlers. Operating Thread in pure star topology is possible but requires running a full mesh stack with mesh routing disabled — retaining all the complexity with none of the topology benefit. Thread also targets AC-powered home automation devices; its power model assumes router-eligible nodes that forward traffic continuously, conflicting with the battery-operated sleepy-node architecture (SYS-C-003). **Eliminated on RTOS dependency (SYS-F-005) and disproportionate stack overhead.**

- **Sigfox:** Sigfox is a proprietary LPWAN requiring a paid operator subscription for both network access and per-device registration. Beyond the commercial dependency, Sigfox has no uplink ACK mechanism — it uses 3× blind repetition instead of acknowledged delivery. Downlink capacity is limited to 4 messages per day at 8 bytes each. With 100+ nodes each requiring per-packet confirmation (SYS-I-005), Sigfox cannot satisfy reliability requirements at any scale. **Eliminated on reliability (no ACK, no retransmission) and proprietary operator dependency.**

- **LoRa (raw PHY):** LoRa is a proprietary chirp-spread-spectrum modulation (Semtech IP) operating at the physical layer only. It provides no addressing, no MAC, no ACK, no retransmission, and no encryption. Any protocol behavior — including requirements SYS-I-005 through SYS-I-009 and SYS-S-005 through SYS-S-013 — must be custom-implemented in firmware, equivalent to building a complete wireless protocol stack from scratch. **Eliminated as raw PHY with no protocol stack.**

- **nRF24L01+:** Nordic's nRF24L01+ is a 2.4 GHz GFSK transceiver with Enhanced ShockBurst — a thin hardware MAC providing auto-ACK and auto-retransmit (up to 15 retries). However, it has no hardware encryption engine; all payloads are transmitted in plaintext, violating SYS-S-005. The hardware architecture further limits each receiver to **6 data pipes**, capping concurrent peer devices at 6. Scaling to 100+ nodes would require custom channel or time-division multiplexing across multiple receivers, reintroducing the protocol complexity that a qualified stack solves natively. **Eliminated on absent encryption and hardware scalability ceiling (6 nodes).**

- **Sub-GHz FSK (raw):** A raw FSK transceiver (e.g., Si4463, SX1231) operates at the physical layer only. Like LoRa PHY, it provides no addressing, no MAC, no ACK, no retransmission, and no encryption. The entire protocol stack must be custom-engineered. **Eliminated as raw PHY with no protocol stack.**

#### Security Observations

**BLE 5.4 (Connected)** provides the strongest link-layer security model: AES-128-CCM encryption with a 32-bit MIC on every Data Channel PDU, replay protection via NESN/SN sequence numbers embedded in the connection context, and mutual authentication via LE Secure Connections (ECDH). Key material (LTK, IRK, CSRK) is negotiated entirely in-protocol per bond. Satisfies SYS-S-005 through SYS-S-013 without any application-layer compensation.

**BLE 5.4 PAwR (Connectionless)** presents a fundamentally different security posture. PAwR operates on the Advertising Physical Channel using Advertising PDUs — not Data Channel PDUs. All BLE link-layer security (AES-CCM encryption, MIC generation, NESN/SN replay tracking, SMP pairing) is defined on the Data Channel and does not apply to Advertising PDUs. The result: PAwR is effectively a raw transport from a security perspective. Encrypted Advertising Data (EAD, BLE 5.3+) can encrypt AD structures using a pre-shared key, but provides no per-packet MIC in the packet header, no authentication of the remote device, and no resistance to replay attacks. Every security requirement (SYS-S-005 through SYS-S-013) must be satisfied at the application layer — equivalent in effort to building a custom security protocol from scratch on top of a raw link.

**Zigbee 3.0** provides a two-layer key hierarchy: a Network Key (shared) encrypting all NWK-layer traffic, and optional Application Link Keys encrypting the APS layer end-to-end between peers. The 32-bit frame counter gives stateless replay protection — no connection context is required, making it robust under intermittent or asymmetric links. Trust Center Install Codes provide a hardware root-of-trust for commissioning (a unique, factory-programmed key embedded in the device join process).

**Zigbee 4.0** (CSA, November 2025) extends 3.0 with three layers of security hardening. First, **cryptographic agility** decouples the security layer from a fixed cipher suite, enabling algorithm upgrades as international standards evolve without breaking backward compatibility with deployed Zigbee 3.0 devices. Second, **key management hardening**: Dynamic Link Key replaces the static pre-configured Link Key — the primary attack surface in 3.0 — with per-session derived keys; Unique Link Key Monitoring detects anomalous key reuse across devices; Trust Center Connectivity guarantees devices can always reach the Trust Center for re-keying or revocation even in degraded network conditions. Third, **network control additions**: Restricted Mode closes commissioning windows after installation, preventing unauthorized joins; Trust Center Swap Out allows coordinator replacement without a full network rekey; PAN ID Changes enable administrators to migrate network identifiers without disconnecting devices. Advanced Frame Counter Synchronization anchors replay counters between specific endpoint pairs rather than globally, eliminating the residual replay window that exists in 3.0 when a device re-joins after a counter reset.

**LoRaWAN** derives NwkSKey and AppSKey from a root AppKey via the OTAA Join Accept procedure (AES-based key derivation). A 32-bit frame counter protects against replay. The primary vulnerability is root AppKey storage in low-cost end devices — a compromised device exposes all derivation material, potentially allowing session key reconstruction. Additionally, the gateway's limited downlink capacity (duty-cycle-constrained) reduces its ability to enforce re-keying or session revocation at scale (100+ nodes).

#### Reliability Observations

**BLE 5.4 (Connected)** provides link-layer ACK via NESN/SN bits embedded in every Data Channel PDU header. A failed packet triggers automatic retransmission within the connection interval, transparent to the application. Connection supervision timeout detects link loss and can trigger reconnection. PDR ≥ 98% is achievable because the connection-based model maintains continuous link state, and every unacknowledged packet is retried automatically.

**BLE 5.4 PAwR** has no guaranteed delivery mechanism at the protocol level. The gateway (broadcaster) allocates response slots per subevent, but there is no ACK transmitted from the broadcaster to the responder confirming receipt. If the gateway misses a response (slot collision, interference), the node has no indication of failure. Retransmission is entirely application-defined — the node must detect the loss via a sequence counter and schedule a retry on the next sync event. PDR ≥ 98% requires careful subevent slot allocation plus application-layer sequence tracking, significantly increasing CPM firmware complexity.

**Zigbee 3.0** uses the IEEE 802.15.4 MAC ACK frame, transmitted by the coordinator within 192 µs (aTurnaroundTime) of receiving a data frame. Retransmission is configurable via `macMaxFrameRetries` (default 3, max 7). For a sleepy end device in star topology: node wakes → transmits → waits for ACK → retransmits if no ACK → sleeps. The entire exchange completes in a few milliseconds and maps directly to the bare-metal super loop wake cycle without additional application-layer machinery.

**Zigbee 4.0** adds four reliability improvements over 3.0. **Expanded APS acknowledgements** introduce an application-support-layer ACK above the MAC ACK, enabling end-to-end delivery confirmation independent of the MAC-layer result — important when the coordinator is proxying for an indirect device. **Standardized network-level retries** add a formal NWK-layer retry mechanism beyond `macMaxFrameRetries`, providing a second retransmission stage without requiring application intervention. **Improved sleepy end device data polling** formalizes the poll-on-wake procedure, reducing the probability of a sleepy device missing its queued message when the parent's indirect TX buffer expires during a prolonged sleep interval. **Formalized Parent Selection** makes the parent-joining decision deterministic under dense deployment conditions, reducing orphan events and reconnection churn. Net result: PDR improvement over 3.0 under interference, dense-node, or intermittent-link scenarios.

**LoRaWAN** Confirmed Uplinks (Class A Confirmed Data Up) receive a downlink ACK from the network server in one of two receive windows (RX1/RX2, typically 1–2 s after uplink). The critical constraint is **duty cycle regulation**: EU868 limits transmitters to 1% on most sub-bands (~36 s/hour). Under 100+ nodes generating concurrent uplinks, the gateway must downlink ACKs for each within its own duty cycle budget. As node density increases, downlink congestion forces ACK delays or drops, triggering retransmit storms. PDR ≥ 98% under 100-node load is not reliably achievable with confirmed messages under EU868 regulatory constraints.

#### Performance & Scalability Observations

**BLE 5.4 (Connected)** scalability ceiling is ~20 simultaneous connections per Central (dependent on the BLE SoC and stack). Serving 100+ nodes requires time-division multiplexing: each node connects → exchanges data → disconnects. Connection setup round trips add 30–100 ms of latency per node under contention. For a low-frequency reporting system (one occupancy update every N seconds), this multiplexing is feasible but adds non-trivial Central firmware complexity.

**BLE 5.4 PAwR** solves the connection ceiling entirely: 128 subevents per sync event, each supporting multiple response slots — hundreds of nodes are serviceable with no connection management. Nodes achieve excellent power efficiency by waking only for their assigned subevent. The costs are the full security deficit documented in §1.1 (requiring an application-layer security stack) and the reliability deficit in §1.2 (requiring application-layer sequence tracking). PAwR's scheduling model is architecturally ideal for power and scalability, but doubles the CPM firmware scope.

**Zigbee 3.0** natively supports ~250 sleepy end devices per coordinator in star topology. The Sleepy End Device role polls the coordinator on wake, transmits, receives the MAC ACK, and returns to sleep — mirroring the bare-metal super loop cycle with no additional state management. Trade-offs: TX current is 10–20 mA (vs. 4–10 mA for BLE) and data rate is 250 kbps, producing longer per-packet air times and marginally higher per-packet energy. The 2.4 GHz band is shared with WiFi and BLE, though DSSS spreading provides some passive interference rejection.

**Zigbee 4.0** introduces three performance-relevant additions over 3.0. **Coordinated Sample Listening (CSL)** enables direct sleepy-to-sleepy communication: two sleepy end devices pre-schedule a wake rendezvous and exchange data without routing through the coordinator, reducing coordinator load and eliminating the two-hop TX energy cost for device-to-device messages. **Batch Commissioning** allows multiple nodes to join the network simultaneously during installation rather than sequentially — directly relevant to deploying 100+ parking nodes in a single session. **Suzi** is the CSA's new sub-GHz brand introduced alongside Zigbee 4.0: built on the same Zigbee NWK layer, it adds the EU 800 MHz and NA 900 MHz PHYs, providing significantly extended range and superior building-material penetration compared to 2.4 GHz. The Suzi certification program is planned to open H1 2026. **Hardware constraint for ParkSense Phase 1:** The STM32WB5MMGH6TR on the B-U585I-IOT02A is a 2.4 GHz-only radio — Suzi requires a different RF module. For an open parking lot (typical span ≤ 300 m), 2.4 GHz Zigbee 3.0 range is sufficient. Suzi becomes relevant for future expansion to covered garages or large-footprint lots.

**LoRaWAN** range (2–15 km) is vastly overprovisioned for a parking lot environment (typical span ≤ 300 m). This range comes at the cost of long air times: at SF12, a 50-byte payload exceeds 2 seconds on-air, with TX current at 20–120 mA — orders of magnitude worse in energy per packet than BLE or Zigbee. The 1% EU868 duty cycle limits a single gateway to fewer than ~40 confirmed uplinks per hour on the most common sub-band, far below the throughput required for 100+ nodes at any useful reporting rate.

## Phase 1 — Protocol Decision

### Selected Protocol: Zigbee 3.0

**Zigbee 3.0** is selected as the primary protocol stack for ParkSense Phase 1.

**Rationale:**
- Satisfies all SyRS requirements without compromise: AES-128 encryption, 32-bit MIC, frame-counter replay protection, Trust Center commissioning, 802.15.4 MAC ACK, configurable retransmission, PDR ≥ 98% achievable, native star topology with ≥ 250 sleepy end devices.
- The STM32WB5MMG module is certified for Bluetooth® LE 5.4, Zigbee® 3.0, and OpenThread, and is included onboard the B-U585I-IOT02A Discovery Kit — no additional RF hardware purchase required for Phase 1.
- Sleepy End Device poll-on-wake model maps directly to the bare-metal cooperative super loop (SYS-F-005): wake → poll → TX → ACK → sleep.
- 2.4 GHz range (100–300 m) is sufficient for an open surface parking lot (typical span ≤ 300 m).

### Future Upgrade Path: Zigbee 4.0

Zigbee 4.0 (CSA, November 2025) delivers meaningful improvements over 3.0 in all three evaluation dimensions — notably Dynamic Link Key, Advanced Frame Counter Synchronization, standardized network-level retries, Batch Commissioning, and Suzi sub-GHz support for covered garages. It is **fully backward compatible** with Zigbee 3.0.

**Why not selected for Phase 1:** At the time of writing, the Suzi certification program has not yet opened (planned H1 2026). Zigbee 4.0–certified modules in accessible hobbyist/prototyping form factors (breakout boards, Feather-compatible, Arduino-shield) are not yet widely available. Selecting Zigbee 4.0 hardware now would constrain the Bill of Materials to non-standard parts and reduce reproducibility for students following this project.

**Upgrade trigger:** If a Zigbee 4.0–certified module with an SPI/UART host interface and open or ST-compatible driver becomes available in the B-U585I-IOT02A ecosystem, upgrading the CPM firmware layer to a 4.0 stack will demonstrate **Layer 3 firmware portability** (SYS-NF-001) — the RF driver is isolated below the CPM API boundary, and the application layer is unchanged.

### Plan B: BLE 5.4 (STM32WB5MMG)

If Zigbee supply or driver support becomes a blocker, the STM32WB5MMG on the same dev kit also supports **BLE 5.4**. Two BLE implementation paths exist:

- **BLE 5.4 Connected + connection multiplexing:** The gateway Central manages a time-division schedule — each node connects, reports, disconnects. Adds ~30–100 ms setup overhead per node but retains full link-layer security (AES-128-CCM, MIC, ECDH). Requires non-trivial Central scheduling logic in the CPM. Maximum ~20 simultaneous connections is managed by ensuring no two nodes overlap.
- **BLE 5.4 PAwR (Connectionless):** Solves the scalability ceiling (hundreds of nodes) and eliminates connection management. Requires implementing AES-128 encryption, HMAC/MIC, and sequence-counter replay protection in the CPM application layer — compensating for the absent link-layer security documented in §1.1. Increases CPM firmware scope but produces a fully custom security stack as a portfolio artifact.

Both BLE paths use the same STM32WB5MMG module — no hardware change is required. The switch is a CPM firmware decision.

## Phase 2 — RF Module Selection

### Selected Module: STM32WB5MMG (STMicroelectronics)

The protocol decision (Zigbee 3.0, with BLE 5.4 as Plan B) determines the module. Both protocols are supported by a single hardware module already present on the development platform.

| Property                  | Detail                                                                 |
| ------------------------- | ---------------------------------------------------------------------- |
| **Module**                | STM32WB5MMGH6TR (SiP — System-in-Package)                              |
| **Die**                   | STM32WB55 dual-core: Cortex-M4 (application) + Cortex-M0+ (RF coprocessor) |
| **Supported stacks**      | Bluetooth® LE 5.4 · Zigbee® 3.0 · OpenThread (IEEE 802.15.4 certified) |
| **Radio**                 | Single 2.4 GHz radio, time-multiplexed by M0+ firmware                 |
| **TX power**              | Up to +6 dBm                                                           |
| **RX sensitivity**        | −96 dBm (BLE 1M) / −100 dBm (802.15.4)                                |
| **Host interface**        | Integrated on-die — M4 and M0+ share SRAM via IPCC mailbox             |
| **Stack delivery**        | Pre-compiled M0+ binary images flashed alongside M4 application        |
| **Dev kit availability**  | Included onboard B-U585I-IOT02A — no separate module purchase required |
| **ST driver/middleware**  | STM32WB Zigbee Middleware (X-CUBE-ZIGBEE), STM32WB BLE stack (CubeWB)  |
| **Certification**         | Bluetooth SIG certified (LE 5.4), CSA Zigbee 3.0 certified             |

**Key integration advantage:** The STM32WB5MMG is co-located on the same PCB as the STM32U585AII6Q application MCU. The M4 RF application core communicates with the STM32U585 application processor via SPI + interrupt lines. This eliminates the external SPI/UART RF module interface that would be required with a discrete radio (e.g., a separate Zigbee breakout board), reducing firmware integration complexity and board-level antenna layout concerns.

**Stack switching:** Changing from Zigbee 3.0 to BLE 5.4 (Plan B) or upgrading to a future Zigbee 4.0 binary requires only flashing a different M0+ firmware image — the M4 application API surface (IPCC mailbox protocol) remains stable. This is the mechanism by which Layer 3 firmware portability (SYS-NF-001) is demonstrated at the RF layer.

## Selection Rationale

**Zigbee 3.0 on STM32WB5MMG** wins on all five evaluation axes:

1. **Security** — AES-128-CCM, 32/64-bit MIC, frame-counter replay protection, Trust Center commissioning, Install Codes. Satisfies SYS-S-005 through SYS-S-013 at the protocol layer with no application-layer compensation.
2. **Reliability** — 802.15.4 MAC ACK within 192 µs, configurable retransmission (0–7×), PDR ≥ 98% achievable. Maps directly to super loop wake/TX/ACK/sleep cycle.
3. **Scalability** — Native star topology with ≥ 250 Sleepy End Devices per coordinator. No connection management, no multiplexing firmware.
4. **Power** — Sleepy End Device polls only on wake. Sleep current < 1 µA. Battery life ≥ 5 years achievable within the node power budget.
5. **Hardware availability** — Module is onboard the B-U585I-IOT02A dev kit. No BOM additions. ST official middleware (X-CUBE-ZIGBEE). Reproducible for students.

## Risk Assessment

| Risk Factor                        | Assessment                                                                                                                                        |
| ---------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------- |
| **ST Zigbee stack (closed binary)**| X-CUBE-ZIGBEE is a proprietary M0+ binary — source not available. Bug fixes depend on ST releases. Mitigated by isolating Zigbee behind CPM API.  |
| **Zigbee 3.0 stack maturity**      | Less community documentation than BLE stack. ST's Zigbee middleware is functional but has fewer public examples than Silicon Labs or NXP stacks.  |
| **2.4 GHz coexistence**            | STM32WB5MMG shares 2.4 GHz with EMW3080B WiFi on the gateway board. PTA (Packet Traffic Arbitration) hardware coexistence available on WB55 die.  |
| **Zigbee 4.0 obsolescence risk**   | Zigbee 3.0 is fully backward compatible with 4.0. No interoperability risk. Stack upgrade is an M0+ binary swap.                                  |
| **Module supply chain**            | STM32WB5MMG is an ST-standard part in production since 2020. Available from Mouser, DigiKey, ST direct. No discontinuation risk in 3–5 year horizon. |
